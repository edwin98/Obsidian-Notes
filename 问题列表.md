# 面试问题列表

## 1. Agent 相关

### 1. Prompt 设计有什么要点？你觉得 Prompt 优化最有效果的方法是什么？

**设计要点（三大原则）：**
- **清晰明确**：明确字数、风格、受众、输出格式，避免模糊指令
- **充足上下文**：告诉模型角色定位（Role-playing）、背景约束、前置条件
- **结构化分隔**：用 XML 标签（`<context>`, `<instructions>`）或 Markdown 将指令、数据、输出要求严格隔开，降低幻觉概率

实战中使用 **CRISPE 框架**（角色/规则/指令/风格/格式/示例），结合 Few-shot CoT 提供带推理过程的示例。

**最有效的优化方法：负面指令转正面 + 关键指令首/尾置。** 例如，不写"不要输出废话"，改写为"只输出 JSON 结构"；核心约束放在 Prompt 开头或结尾（Lost in the Middle 现象）。

在项目中，Query 改写 Prompt 的 System Prompt 专门约束了 JSON 输出格式并给出 Mock 示例，避免模型输出自由文本导致后续解析崩溃。

---

### 2. Human in the loop 是如何做的？权限问题、拒绝回答、澄清等人工介入如何实现？

**工程难点**：不能用 while 循环等待审批，会导致线程阻塞和 HTTP 504 超时。

**实现方案**（基于 LangGraph）：
1. 在高风险节点外侧挂 `interrupt_before` 钩子，触发后 LangGraph 将完整 State 序列化快照到 Postgres（Checkpoint），子进程终止、显存释放
2. Celery 异步触发钉钉 Webhook 通知专家
3. 专家在 Web 前端 Review 用例，修改危险参数后点击 Approve
4. 前端调用唤醒 API，后端从 Postgres 按 Thread_ID 拉取 State，注入修改内容，调用 `.resume()` 恢复执行

**三类触发场景**：
1. **灾难性高危指令拦截**：拦截包含高危写权限信令的用例（如 `format_c`, `reset_all`）
2. **置信度灰度区间**（0.60~0.65）：模型自爆不确定性，交由专家会诊
3. **全新低相似度 Novel Case 确权**：LLM "灵光一闪"生成的用例与知识库相似度极低，需专家鉴别价值

---

### 3. 如何进行记忆隔离？有什么场景需要用到长期记忆吗？

**记忆隔离**：通过 LangGraph 的 Thread_ID 机制实现会话级硬隔离。每个用户/任务有独立 Thread_ID，Postgres Checkpoint 存储各线程的完整 State，并发多租户场景下完全不会串键。

**长期记忆的场景**：
- **历史缺陷库**（5G Agent）：将历史 Bugzilla 缺陷 Embedding 入 Milvus，生成用例时防止遗漏已知边界场景
- **Golden Cases 知识库**：专家确认的高质量用例回流入库，数据飞轮越用越聪明
- **RAG 系统的多轮对话**：上轮对话结论作为 Observation 注入下一轮，避免重复检索

**短期记忆的压缩策略**：滑动窗口截断 + 超出 Token 预算时 Celery 异步调用 4B 小模型进行摘要压缩，更新 State。

---

### 4. 怎么去测试 Agent 的好坏？怎么去构造 Ground Truth？怎么评估 Agent 的效果？与传统软件测试有什么区别？

**四维评估矩阵**：

| 维度 | 指标 | 方法 |
|------|------|------|
| 格式合规 | JSON Schema 合法率（指令遵从率） | pytest + Pydantic Validators |
| 工具调用 | 工具选择准确率 + 参数 F1-Score | 与 Ground Truth 轨迹对比 |
| 轨迹效率 | 成功率 + 步数效率比（Agent步数/专家步数） | Golden Trajectories 对比 |
| 安全合规 | 高危阻断率（生产网要求 100%） | 毒药测试集压测 |

**三层评估技术栈**：
1. pytest + Ragas/DeepEval 量化检索精度
2. LLM-as-Judge（Prometheus-Eval/GPT-4o）对长文本语义评分
3. 沙盒执行层（最关键）：用例直接在 Docker 隔离环境运行，以执行结果而非文本描述为最终判定

**Ground Truth 构造**：从 HITL 沉淀的专家确认数据中提取，积累约 2 万条 Golden Dataset。

**与传统软件测试的区别**：传统测试有确定性的输入输出，可以精确断言；Agent 的中间步骤是概率性的，同一个目标可以有多条有效路径，因此需要轨迹级评估而非单次输出对比。Ground Truth 不是某个固定答案，而是一组专家认可的"合格轨迹集合"（Golden Trajectories）。

---

### 5. 什么是 Prompt 遵从性？怎么提高 Agent 的 Prompt 遵从性？如何提高 Agent 的稳定性？

**Prompt 遵从性（Instruction Following）**：模型按照 Prompt 中明确指令执行的比率，包括格式遵从（输出合法 JSON）、规则遵从（不输出高危内容）、角色遵从（保持专家视角）。

**提高方法**：
- **SFT 微调**：通过有监督微调，将"思考过程 + 严格 JSON 工具调用语法"的规范烙印在模型权重中，比纯 Prompt 稳定得多
- **结构化输出约束**：使用 Function Calling / Tool Use 原生支持，模型直接输出结构化 JSON，而非让正则解析自由文本
- **Guardrail 双缝检测**：正则规则库 + 轻量分类器对输出进行后置校验，不符合格式时重试或降级

**提高稳定性**：双重熔断机制（硬迭代上限 `recursion_limit=15` + 置信度软熔断 `< 0.6` 触发 HITL）防止死循环和失控。

---

### 6. Function Calling 和 MCP 有什么区别？两者是什么关系？

**Function Calling**（也叫 Tool Use）是 LLM 的一种能力：模型在推理时决定调用哪个工具并输出结构化参数，运行时由 Agent 框架执行实际函数调用后将结果作为 Observation 返回。这是 Agent-工具交互的**协议层**。

**MCP（Model Context Protocol）** 是 Anthropic 提出的一套标准化协议，解决"工具定义如何统一描述、如何被不同模型和框架消费"的问题，是工具生态的**传输/注册标准层**。

**关系**：MCP 是对 Function Calling 的规范化和标准化封装。Function Calling 是具体的调用行为，MCP 定义了工具如何被发现、描述、注册和调用的通用规范——类似 HTTP 之于 Web 请求。

---

### 7. ReAct 耗时问题：怎么终止退出思考？怎么避免思考死循环？

**终止机制**：
- **硬终止**：LangGraph `recursion_limit`（如 15 跳）强制挂起，防止无限 Thought-Action-Observation 循环
- **软终止**：强制 LLM 输出 `confidence_score`，当连续两次置信度 < 0.6 时触发降级，进入 HITL 而非继续自转

**避免死循环**：
- 检测"重复 Action"：若连续两次 Action 完全相同（参数相同），判定为陷入循环，直接退出
- 工具调用错误计数（`error_count`）超阈值后强制跳出
- Guardrail 层检测输出是否在"幻觉循环"状态

---

### 8. 讲一下你们 Agent 的架构以及包含的组件，为什么要这么设计？

**整体架构**：FastAPI 接口层 → LangGraph 状态机引擎 → 工具层 → 基础设施层

**关键组件**：
- **意图解析节点**：识别测试需求类型，决定走哪条分支
- **用例生成节点**（Qwen3-32B）：RAG 增强生成测试用例
- **Guardrail 护栏**：正则 + 轻量分类器双缝检测高危意图
- **执行节点**：调用仿真环境工具（Simulation_Runner 等）
- **HITL 节点**：LangGraph Checkpoint + Celery Webhook 实现无阻塞人工介入
- **结果判定节点**：统计轨（KPI 假设检验）+ 语义轨（LLM 日志分析）双轨并行

**设计理由**：状态机比纯 ReAct 更可控——明确的节点和边定义使得测试、调试、HITL 拦截都有清晰的切入点；Checkpoint 持久化解决了长任务的断点续跑；双重熔断保证了生产安全。

---

### 9. 框架选型：为什么选择 LangGraph 而不选择其他的（比如 AutoGen、LangChain 等）？LangGraph 和 LangChain 是什么区别？

**LangGraph vs LangChain**：LangChain 是工具链库，提供工具抽象、LLM 封装、LCEL（链式组合）；LangGraph 是建立在 LangChain 基础上的**有向图状态机引擎**，专门解决"需要循环、条件分支、持久化状态的复杂 Agent"问题。两者不是替代关系，LangGraph 依赖 LangChain 的基础组件。

**为什么不选 AutoGen**：AutoGen 擅长多 Agent 对话，但状态管理和 Checkpoint 持久化不够精细；对于"需要 HITL 人工介入 + 跨会话断点续跑"的工业场景，LangGraph 的原生 Checkpoint 支持更完备。

**为什么选 LangGraph**：
1. 原生支持 `interrupt_before/after`，HITL 实现优雅
2. Postgres Saver 实现状态持久化，支持跨进程的任务恢复
3. 图结构使并行节点（Map-Reduce 多 Agent）和条件路由清晰可维护
4. 生态成熟，与 LangSmith 集成方便追踪调试

---

### 10. ToB 场景下设计 Agent 架构，会主要从哪些方面去考虑？

1. **安全性优先**：Guardrail 护栏 + HITL 人工拦截，高危场景必须人工确权，不能让 Agent 完全自主
2. **可解释性**：每一步的 Thought/Action/Observation 完整记录，专家可审计；Confidence Score 量化不确定性
3. **可控的状态管理**：Checkpoint 持久化支持断点续跑，避免长任务网络中断导致全部重来
4. **性能与成本**：主干推理用大模型（32B），意图分类/格式校验等非核心节点用 SFT 后的小模型（4B）降低延迟和成本；vLLM 提升吞吐
5. **权限与多租户隔离**：RBAC + Thread_ID 隔离，SAP 画像做标量过滤，不同人员/组织看到不同知识范围
6. **可观测性**：SSE 流式推送让前端实时展示 Agent 执行进度，不是黑箱

---

### 11. MCP 通信协议为什么需要 SSE 和 Message 两个端点？一个不行吗？

MCP 采用两个独立端点是为了分离**推送通道**和**请求通道**的职责：

- **SSE 端点**（服务端推送）：服务器主动向客户端推送工具注册信息、执行进度和结果流，客户端无需轮询，低延迟
- **Message 端点**（客户端请求）：客户端向服务器发送工具调用请求

这种设计本质是**半双工分离**——HTTP 原生是请求-响应模型，无法在同一连接上同时满足"服务端主动推"和"客户端主动发"。用一个端点的话要么退化为轮询（高延迟），要么改用 WebSocket（但 SSE+HTTP POST 更简单、更兼容代理/CDN）。这也是项目 FastAPI 接口层选择 SSE 推流给前端展示实时状态的原因。

### 12. 业界测试 Agent 能力的时候有哪些 Benchmark？

- **HotpotQA / MuSiQue**：多跳推理，测试 Agent 跨文档关联能力
- **WebArena / WorkArena**：Web 浏览器操作任务，测试 GUI Agent 的实际执行能力

- **GAIA**：通用 AI 助手能力测评，涵盖工具使用、文件处理、多步推理

- **τ-bench**：工具调用准确性专项测试

项目中自研了针对 5G 测试场景的内部 Benchmark：用毒药测试集（高危用例）测安全阻断率，用 Golden Trajectories 测步数效率比，用 Pydantic 校验测格式合规率。

### 13. Agent 记忆管理如何进行剪枝和压缩？

**压缩策略**：

1. **滑动窗口截断**：只保留最近 N 轮对话作为短期记忆，超出部分丢弃或压缩
2. **定期摘要压缩（Summary）**：LangGraph 监测 Token 预算，溢出时 Celery 异步触发 4B 小模型对历史轮次进行摘要浓缩，替换原始消息列表，更新图节点状态
3. **重要性剪枝**：工具执行的原始 Observation 往往很长，提取关键字段（如 KPI 指标值）后丢弃原始日志
4. **长期记忆向量化**：高价值结论（专家确认的 Golden Case、HITL 沉淀数据）Embedding 入 Milvus，脱离上下文窗口限制

---

### 14. 如何衡量 Multi-agent 的性能和准确率？评估方法是怎样的？

**性能指标**：
- 端到端任务完成率（E2E Success Rate）
- 各子 Agent 的局部完成率（便于定位瓶颈节点）
- 总步数效率比（Agent 步数 / 专家步数）
- 协调开销（消息传递次数、等待时延）

**准确率评估**：
- 沙盒执行（最权威）：最终输出在隔离环境中执行，以真实结果判定正确性
- LLM-as-Judge：对子 Agent 的中间输出进行语义评分
- 与 Golden Trajectories 对比：每个子 Agent 的输出与专家确认路径的 F1-Score

**Map-Reduce 多 Agent 实践**：主控 Agent 分发任务给三个 Worker（分别分析 RRC 信令、NAS 信令、KPI），各自输出分析结论，汇总节点 Reduce 合并为诊断报告。评估时除 E2E 准确率外，还分别评估每个 Worker 的召回率，避免某一路 Worker 成为整体瓶颈。

## 2. RAG 相关

### 1. RAG 的检索准确性如何提升？做了哪些方面的优化？

端到端 E2E 准确率最终达到 **93%**，核心优化手段：

1. **三级渐进式召回**：第一层（BM25+384维向量）召回率 99%，逐层精筛至 Top 10，兼顾覆盖与精度
2. **Embedding 模型领域微调**：基于 gte-multilingual-base 微调，75 万 QA 对训练，自有数据集 Top10 召回率从 62% 提升到 88%
3. **问题改写（Query Rewrite）**：指代消解 + 问题扩展，将一个问题扩展为 1~3 个子问题并行检索
4. **RSF 动态权重融合**：根据 Query Token 数量动态调节向量/BM25 权重比例
5. **父子索引**：子切片精确命中后回溯父文档，提供完整上下文
6. **专有词表 + 定制分词器**：6800 个无线领域词汇，防止"随机接入"等术语被错误切分

---

### 2. 不同文档类型的分割方式应该不一样，在这个方面有做限制吗？如何评估文档切分和检索的效果？

**差异化切分策略**：

| 格式 | 处理方式 |
|------|----------|
| Word / HTML / PDF | 统一转 Markdown，按层级结构（标题层次）树形切分 |
| PPT | 按页转图片 → OCR → 生成图片链接，不参与检索只参与生成 |
| iCase（反思案例）、华为案例 | 不切片，整篇入库 |
| 非叶子节点（含子章节）> 2K Token | LLM 生成摘要替代 |
| 叶子节点 | 512~800 Token 固定步长，10%~15% 重叠率 |

**评估方式**：
- 构建评测集（2034 条），涵盖三种提问方式（原文/总结/容错）
- 三个等级：正确召回、部分召回、无召回
- 重点指标：Recall@K（K=10/20/80）
- 父子索引检验：命中子切片后能否完整恢复父文档上下文

---

### 3. RAG 知识库用的什么向量库？体量有多大？Milvus 索引用的哪种类型？为什么会选择 IVF 索引，而不是 HNSW 索引？

使用 **Milvus**，知识库体量为**千万级** Chunk，使用 **HNSW 索引**（问题中 IVF 和 HNSW 倒置了，实际选的是 HNSW）。

**为什么选 HNSW 而非 IVF**：

| | HNSW | IVF |
|--|------|-----|
| 查询方式 | 图遍历，近似最近邻 | 先量化分区再搜索 |
| 查询延迟 | 极低（毫秒级，无需预热） | 需要加载倒排列表，有 IO 开销 |
| 召回率 | 高，受 `ef` 参数控制 | 受 `nprobe` 参数控制 |
| 内存占用 | 较高（图结构需常驻内存） | 较低（支持磁盘存储） |
| 适用场景 | 在线实时检索，低延迟要求 | 超大规模离线场景，内存受限 |

系统是在线检索场景，要求检索延迟控制在毫秒级，HNSW 更合适。如果是离线批量相似搜索且内存受限，才会考虑 IVF_FLAT 或 IVF_PQ。

---

### 4. Milvus 全文检索用的什么分词器？有比较过区别吗？

BM25 这路走的是 **Elasticsearch + 定制 IK 分词器**，挂载了内部 6800 词的无线领域专有词表。相比 jieba 默认词典，定制词表能正确识别"随机接入"、"载波聚合"、"RSRP"等术语，不会被切成无意义片段。

ES 和 Milvus 的 BM25 核心区别：ES 的分词器生态更成熟，支持自定义词典热更新；Milvus 的全文检索是后加的功能，词典扩展灵活性相对弱。因此将 BM25 检索放在 ES，向量检索放在 Milvus，两者并行。

**词表构建流程**：汇集 DTS 工单 + Support 文档 + IDP 文档 → 爬取华为术语库（5000 词种子）→ AutoPhrase 自动发现新词（约 10000 词）→ Qwen2.5-72B 打分过滤 → 最终 6800 个有效词。

---

### 5. BM25 的检索过程是怎样的？k、b 有什么用？

BM25 打分公式：

$$Score(Q, D) = \sum_{i=1}^{n} IDF(q_i) \cdot \frac{TF(q_i, D) \cdot (k_1 + 1)}{TF(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{avgdl})}$$

**检索过程**：对 Query 分词，对每个词查倒排索引找到包含该词的文档，按上述公式对候选文档逐词求和打分，排序返回 Top-K。

**k1 的作用**：控制词频饱和度。k1 越大，词频越高的文档得分增加越显著；k1 越小，单词多次出现的边际收益越低。通常设 1.2~2.0。

**b 的作用**：控制文档长度归一化强度。b=1 时完全按长度归一化，长文档中命中词得分被大幅压缩；b=0 时不做长度惩罚。通常设 0.75，平衡长短文档的竞争公平性。

### 6. RAG 检索如何加速？

1. **384 维轻量向量做粗筛**：粗筛阶段用低维向量（计算量小），减少全量比较开销
2. **HNSW 索引**：图遍历近似检索，避免全量暴力扫描
3. **候选集复用**：第二层 BM25 分数直接复用第一层缓存，不重新检索
4. **并发双路检索**：BM25 和向量检索并发请求，不串行等待
5. **Reranker 仅处理 80 条**：精排输入量控制在可接受范围，整体三级链路约 2s
6. **Kafka 异步解耦**：离线入库走 Kafka，不影响在线检索链路

### 7. 如何衡量检索质量？召回率怎么计算？如何判别召回文档是符合标准的？这个标准谁提供？

**召回率计算**：

$$Recall@K = \frac{\text{前 K 个结果中包含正确文档的数量}}{\text{该问题相关文档总数}}$$

**三个等级**：正确召回（完整命中）、部分召回（命中了相关 Chunk 但不完整）、无召回。

**判别标准来源**：
- 自研评测集（2034 条），由领域专家标注每个问题的正确答案所在文档（Ground Truth）
- 三种提问方式（原文/总结/容错）覆盖不同召回难度
- 对于 Agent 支撑场景，额外评估"Agent 作业上下文获取"——即 Agent 在任务执行中能否精准拿到所需上下文

### 8. 在什么时候应该挂 RAG 知识库，什么时候需要进行模型微调？

| 场景 | 选 RAG | 选微调 |
|------|--------|--------|
| 知识时效性强、频繁更新 | ✓（新文档入库即可） | 重新训练成本高 |
| 需要引用来源、可溯源 | ✓（可标注 Chunk 出处） | 模型参数内无法溯源 |
| 领域术语理解不足 | ✗（RAG 只是上下文，模型本身不理解） | ✓（SFT 将术语概念烙印权重） |
| 格式/结构化输出不稳定 | ✗（Prompt 效果有限） | ✓（SFT 将格式肌肉记忆烙印） |
| 需要抑制危险输出倾向 | ✗（护栏是后置拦截，治标） | ✓（DPO 对齐治本） |
| 推理延迟要求极低 | ✗（多了检索环节） | ✓（推理阶段无额外开销） |

实践：两者结合——RAG 解决知识时效性和可溯源性；SFT 解决领域术语理解和格式遵从；DPO 解决危险指令幻觉抑制。三者互补，分别针对不同问题根因。
