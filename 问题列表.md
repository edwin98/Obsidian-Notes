# 面试问题列表

## 1. Agent 相关

### 1. Prompt 设计有什么要点？你觉得 Prompt 优化最有效果的方法是什么？

**设计要点（三大原则）：**
- **清晰明确**：明确字数、风格、受众、输出格式，避免模糊指令
- **充足上下文**：告诉模型角色定位（Role-playing）、背景约束、前置条件
- **结构化分隔**：用 XML 标签（`<context>`, `<instructions>`）或 Markdown 将指令、数据、输出要求严格隔开，降低幻觉概率

实战中使用 **CRISPE 框架**（角色/规则/指令/风格/格式/示例），结合 Few-shot CoT 提供带推理过程的示例。

**最有效的优化方法：负面指令转正面 + 关键指令首/尾置。** 例如，不写"不要输出废话"，改写为"只输出 JSON 结构"；核心约束放在 Prompt 开头或结尾（Lost in the Middle 现象）。

在项目中，Query 改写 Prompt 的 System Prompt 专门约束了 JSON 输出格式并给出 Mock 示例，避免模型输出自由文本导致后续解析崩溃。

---

### 2. Human in the loop 是如何做的？权限问题、拒绝回答、澄清等人工介入如何实现？

**工程难点**：不能用 while 循环等待审批，会导致线程阻塞和 HTTP 504 超时。

**实现方案**（基于 LangGraph）：
1. 在高风险节点外侧挂 `interrupt_before` 钩子，触发后 LangGraph 将完整 State 序列化快照到 Postgres（Checkpoint），子进程终止、显存释放
2. Celery 异步触发钉钉 Webhook 通知专家
3. 专家在 Web 前端 Review 用例，修改危险参数后点击 Approve
4. 前端调用唤醒 API，后端从 Postgres 按 Thread_ID 拉取 State，注入修改内容，调用 `.resume()` 恢复执行

**三类触发场景**：
1. **灾难性高危指令拦截**：拦截包含高危写权限信令的用例（如 `format_c`, `reset_all`）
2. **置信度灰度区间**（0.60~0.65）：模型自爆不确定性，交由专家会诊
3. **全新低相似度 Novel Case 确权**：LLM "灵光一闪"生成的用例与知识库相似度极低，需专家鉴别价值

---

### 3. 如何进行记忆隔离？有什么场景需要用到长期记忆吗？

**记忆隔离**：通过 LangGraph 的 Thread_ID 机制实现会话级硬隔离。每个用户/任务有独立 Thread_ID，Postgres Checkpoint 存储各线程的完整 State，并发多租户场景下完全不会串键。

**长期记忆的场景**：
- **历史缺陷库**（5G Agent）：将历史 Bugzilla 缺陷 Embedding 入 Milvus，生成用例时防止遗漏已知边界场景
- **Golden Cases 知识库**：专家确认的高质量用例回流入库，数据飞轮越用越聪明
- **RAG 系统的多轮对话**：上轮对话结论作为 Observation 注入下一轮，避免重复检索

**短期记忆的压缩策略**：滑动窗口截断 + 超出 Token 预算时 Celery 异步调用 4B 小模型进行摘要压缩，更新 State。

---

### 4. 怎么去测试 Agent 的好坏？怎么去构造 Ground Truth？怎么评估 Agent 的效果？与传统软件测试有什么区别？

**四维评估矩阵**：

| 维度 | 指标 | 方法 |
|------|------|------|
| 格式合规 | JSON Schema 合法率（指令遵从率） | pytest + Pydantic Validators |
| 工具调用 | 工具选择准确率 + 参数 F1-Score | 与 Ground Truth 轨迹对比 |
| 轨迹效率 | 成功率 + 步数效率比（Agent步数/专家步数） | Golden Trajectories 对比 |
| 安全合规 | 高危阻断率（生产网要求 100%） | 毒药测试集压测 |

**三层评估技术栈**：
1. pytest + Ragas/DeepEval 量化检索精度
2. LLM-as-Judge（Prometheus-Eval/GPT-4o）对长文本语义评分
3. 沙盒执行层（最关键）：用例直接在 Docker 隔离环境运行，以执行结果而非文本描述为最终判定

**Ground Truth 构造**：从 HITL 沉淀的专家确认数据中提取，积累约 2 万条 Golden Dataset。

**与传统软件测试的区别**：传统测试有确定性的输入输出，可以精确断言；Agent 的中间步骤是概率性的，同一个目标可以有多条有效路径，因此需要轨迹级评估而非单次输出对比。Ground Truth 不是某个固定答案，而是一组专家认可的"合格轨迹集合"（Golden Trajectories）。

---

### 5. 什么是 Prompt 遵从性？怎么提高 Agent 的 Prompt 遵从性？如何提高 Agent 的稳定性？

**Prompt 遵从性（Instruction Following）**：模型按照 Prompt 中明确指令执行的比率，包括格式遵从（输出合法 JSON）、规则遵从（不输出高危内容）、角色遵从（保持专家视角）。

**提高方法**：
- **SFT 微调**：通过有监督微调，将"思考过程 + 严格 JSON 工具调用语法"的规范烙印在模型权重中，比纯 Prompt 稳定得多
- **结构化输出约束**：使用 Function Calling / Tool Use 原生支持，模型直接输出结构化 JSON，而非让正则解析自由文本
- **Guardrail 双缝检测**：正则规则库 + 轻量分类器对输出进行后置校验，不符合格式时重试或降级

**提高稳定性**：双重熔断机制（硬迭代上限 `recursion_limit=15` + 置信度软熔断 `< 0.6` 触发 HITL）防止死循环和失控。

---

### 6. Function Calling 和 MCP 有什么区别？两者是什么关系？

**Function Calling**（也叫 Tool Use）是 LLM 的一种能力：模型在推理时决定调用哪个工具并输出结构化参数，运行时由 Agent 框架执行实际函数调用后将结果作为 Observation 返回。这是 Agent-工具交互的**协议层**。

**MCP（Model Context Protocol）** 是 Anthropic 提出的一套标准化协议，解决"工具定义如何统一描述、如何被不同模型和框架消费"的问题，是工具生态的**传输/注册标准层**。

**关系**：MCP 是对 Function Calling 的规范化和标准化封装。Function Calling 是具体的调用行为，MCP 定义了工具如何被发现、描述、注册和调用的通用规范——类似 HTTP 之于 Web 请求。

---

### 7. ReAct 耗时问题：怎么终止退出思考？怎么避免思考死循环？

**终止机制**：
- **硬终止**：LangGraph `recursion_limit`（如 15 跳）强制挂起，防止无限 Thought-Action-Observation 循环
- **软终止**：强制 LLM 输出 `confidence_score`，当连续两次置信度 < 0.6 时触发降级，进入 HITL 而非继续自转

**避免死循环**：
- 检测"重复 Action"：若连续两次 Action 完全相同（参数相同），判定为陷入循环，直接退出
- 工具调用错误计数（`error_count`）超阈值后强制跳出
- Guardrail 层检测输出是否在"幻觉循环"状态

---

### 8. 讲一下你们 Agent 的架构以及包含的组件，为什么要这么设计？

**整体架构**：FastAPI 接口层 → LangGraph 状态机引擎 → 工具层 → 基础设施层

**关键组件**：
- **意图解析节点**：识别测试需求类型，决定走哪条分支
- **用例生成节点**（Qwen3-32B）：RAG 增强生成测试用例
- **Guardrail 护栏**：正则 + 轻量分类器双缝检测高危意图
- **执行节点**：调用仿真环境工具（Simulation_Runner 等）
- **HITL 节点**：LangGraph Checkpoint + Celery Webhook 实现无阻塞人工介入
- **结果判定节点**：统计轨（KPI 假设检验）+ 语义轨（LLM 日志分析）双轨并行

**设计理由**：状态机比纯 ReAct 更可控——明确的节点和边定义使得测试、调试、HITL 拦截都有清晰的切入点；Checkpoint 持久化解决了长任务的断点续跑；双重熔断保证了生产安全。

---

### 9. 框架选型：为什么选择 LangGraph 而不选择其他的（比如 AutoGen、LangChain 等）？LangGraph 和 LangChain 是什么区别？

**LangGraph vs LangChain**：LangChain 是工具链库，提供工具抽象、LLM 封装、LCEL（链式组合）；LangGraph 是建立在 LangChain 基础上的**有向图状态机引擎**，专门解决"需要循环、条件分支、持久化状态的复杂 Agent"问题。两者不是替代关系，LangGraph 依赖 LangChain 的基础组件。

**为什么不选 AutoGen**：AutoGen 擅长多 Agent 对话，但状态管理和 Checkpoint 持久化不够精细；对于"需要 HITL 人工介入 + 跨会话断点续跑"的工业场景，LangGraph 的原生 Checkpoint 支持更完备。

**为什么选 LangGraph**：
1. 原生支持 `interrupt_before/after`，HITL 实现优雅
2. Postgres Saver 实现状态持久化，支持跨进程的任务恢复
3. 图结构使并行节点（Map-Reduce 多 Agent）和条件路由清晰可维护
4. 生态成熟，与 LangSmith 集成方便追踪调试

---

### 10. ToB 场景下设计 Agent 架构，会主要从哪些方面去考虑？

1. **安全性优先**：Guardrail 护栏 + HITL 人工拦截，高危场景必须人工确权，不能让 Agent 完全自主
2. **可解释性**：每一步的 Thought/Action/Observation 完整记录，专家可审计；Confidence Score 量化不确定性
3. **可控的状态管理**：Checkpoint 持久化支持断点续跑，避免长任务网络中断导致全部重来
4. **性能与成本**：主干推理用大模型（32B），意图分类/格式校验等非核心节点用 SFT 后的小模型（4B）降低延迟和成本；vLLM 提升吞吐
5. **权限与多租户隔离**：RBAC + Thread_ID 隔离，SAP 画像做标量过滤，不同人员/组织看到不同知识范围
6. **可观测性**：SSE 流式推送让前端实时展示 Agent 执行进度，不是黑箱

---

### 11. MCP 通信协议为什么需要 SSE 和 Message 两个端点？一个不行吗？

MCP 采用两个独立端点是为了分离**推送通道**和**请求通道**的职责：

- **SSE 端点**（服务端推送）：服务器主动向客户端推送工具注册信息、执行进度和结果流，客户端无需轮询，低延迟
- **Message 端点**（客户端请求）：客户端向服务器发送工具调用请求

这种设计本质是**半双工分离**——HTTP 原生是请求-响应模型，无法在同一连接上同时满足"服务端主动推"和"客户端主动发"。用一个端点的话要么退化为轮询（高延迟），要么改用 WebSocket（但 SSE+HTTP POST 更简单、更兼容代理/CDN）。这也是项目 FastAPI 接口层选择 SSE 推流给前端展示实时状态的原因。

### 12. 业界测试 Agent 能力的时候有哪些 Benchmark？

- **HotpotQA / MuSiQue**：多跳推理，测试 Agent 跨文档关联能力
- **WebArena / WorkArena**：Web 浏览器操作任务，测试 GUI Agent 的实际执行能力
- **SWE-bench**：软件工程任务（GitHub Issues），测试代码 Agent 的真实解题率
- **GAIA**：通用 AI 助手能力测评，涵盖工具使用、文件处理、多步推理
- **AgentBench**：多环境（操作系统、数据库、Web）综合能力测评
- **τ-bench**：工具调用准确性专项测试

项目中自研了针对 5G 测试场景的内部 Benchmark：用毒药测试集（高危用例）测安全阻断率，用 Golden Trajectories 测步数效率比，用 Pydantic 校验测格式合规率。

### 13. Agent 记忆管理如何进行剪枝和压缩？

**压缩策略**：

1. **滑动窗口截断**：只保留最近 N 轮对话作为短期记忆，超出部分丢弃或压缩
2. **定期摘要压缩（Summary）**：LangGraph 监测 Token 预算，溢出时 Celery 异步触发 4B 小模型对历史轮次进行摘要浓缩，替换原始消息列表，更新图节点状态
3. **重要性剪枝**：工具执行的原始 Observation 往往很长，提取关键字段（如 KPI 指标值）后丢弃原始日志
4. **长期记忆向量化**：高价值结论（专家确认的 Golden Case、HITL 沉淀数据）Embedding 入 Milvus，脱离上下文窗口限制

---

### 14. 如何衡量 Multi-agent 的性能和准确率？评估方法是怎样的？

**性能指标**：
- 端到端任务完成率（E2E Success Rate）
- 各子 Agent 的局部完成率（便于定位瓶颈节点）
- 总步数效率比（Agent 步数 / 专家步数）
- 协调开销（消息传递次数、等待时延）

**准确率评估**：
- 沙盒执行（最权威）：最终输出在隔离环境中执行，以真实结果判定正确性
- LLM-as-Judge：对子 Agent 的中间输出进行语义评分
- 与 Golden Trajectories 对比：每个子 Agent 的输出与专家确认路径的 F1-Score

**Map-Reduce 多 Agent 实践**：主控 Agent 分发任务给三个 Worker（分别分析 RRC 信令、NAS 信令、KPI），各自输出分析结论，汇总节点 Reduce 合并为诊断报告。评估时除 E2E 准确率外，还分别评估每个 Worker 的召回率，避免某一路 Worker 成为整体瓶颈。

## 2. RAG 相关

### 1. RAG 的检索准确性如何提升？做了哪些方面的优化？

端到端 E2E 准确率最终达到 **93%**，核心优化手段：

1. **三级渐进式召回**：第一层（BM25+384维向量）召回率 99%，逐层精筛至 Top 10，兼顾覆盖与精度
2. **Embedding 模型领域微调**：基于 gte-multilingual-base 微调，75 万 QA 对训练，自有数据集 Top10 召回率从 62% 提升到 88%
3. **问题改写（Query Rewrite）**：指代消解 + 问题扩展，将一个问题扩展为 1~3 个子问题并行检索
4. **RSF 动态权重融合**：根据 Query Token 数量动态调节向量/BM25 权重比例
5. **父子索引**：子切片精确命中后回溯父文档，提供完整上下文
6. **专有词表 + 定制分词器**：6800 个无线领域词汇，防止"随机接入"等术语被错误切分

---

### 2. 不同文档类型的分割方式应该不一样，在这个方面有做限制吗？如何评估文档切分和检索的效果？

**差异化切分策略**：

| 格式 | 处理方式 |
|------|----------|
| Word / HTML / PDF | 统一转 Markdown，按层级结构（标题层次）树形切分 |
| PPT | 按页转图片 → OCR → 生成图片链接，不参与检索只参与生成 |
| iCase（反思案例）、华为案例 | 不切片，整篇入库 |
| 非叶子节点（含子章节）> 2K Token | LLM 生成摘要替代 |
| 叶子节点 | 512~800 Token 固定步长，10%~15% 重叠率 |

**评估方式**：
- 构建评测集（2034 条），涵盖三种提问方式（原文/总结/容错）
- 三个等级：正确召回、部分召回、无召回
- 重点指标：Recall@K（K=10/20/80）
- 父子索引检验：命中子切片后能否完整恢复父文档上下文

### 3. RAG 知识库用的什么向量库？体量有多大？Milvus 索引用的哪种类型？为什么会选择 IVF 索引，而不是 HNSW 索引？

使用 **Milvus**，知识库体量为**千万级** Chunk，使用 **HNSW 索引**（问题中 IVF 和 HNSW 倒置了，实际选的是 HNSW）。

**为什么选 HNSW 而非 IVF**：

|      | HNSW          | IVF              |
| ---- | ------------- | ---------------- |
| 查询方式 | 图遍历，近似最近邻     | 先量化分区再搜索         |
| 查询延迟 | 极低（毫秒级，无需预热）  | 需要加载倒排列表，有 IO 开销 |
| 召回率  | 高，受 `ef` 参数控制 | 受 `nprobe` 参数控制  |
| 内存占用 | 较高（图结构需常驻内存）  | 较低（支持磁盘存储）       |
| 适用场景 | 在线实时检索，低延迟要求  | 超大规模离线场景，内存受限    |

系统是在线检索场景，要求检索延迟控制在毫秒级，HNSW 更合适。如果是离线批量相似搜索且内存受限，才会考虑 IVF_FLAT 或 IVF_PQ。

---

### 4. Milvus 全文检索用的什么分词器？有比较过区别吗？

BM25 这路走的是 **Elasticsearch + 定制 IK 分词器**，挂载了内部 6800 词的无线领域专有词表。相比 jieba 默认词典，定制词表能正确识别"随机接入"、"载波聚合"、"RSRP"等术语，不会被切成无意义片段。

ES 和 Milvus 的 BM25 核心区别：ES 的分词器生态更成熟，支持自定义词典热更新；Milvus 的全文检索是后加的功能，词典扩展灵活性相对弱。因此将 BM25 检索放在 ES，向量检索放在 Milvus，两者并行。

**词表构建流程**：汇集 DTS 工单 + Support 文档 + IDP 文档 → 爬取华为术语库（5000 词种子）→ AutoPhrase 自动发现新词（约 10000 词）→ Qwen2.5-72B 打分过滤 → 最终 6800 个有效词。

### 5. BM25 的检索过程是怎样的？k、b 有什么用？

BM25 打分公式：

$$Score(Q, D) = \sum_{i=1}^{n} IDF(q_i) \cdot \frac{TF(q_i, D) \cdot (k_1 + 1)}{TF(q_i, D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{avgdl})}$$

**检索过程**：对 Query 分词，对每个词查倒排索引找到包含该词的文档，按上述公式对候选文档逐词求和打分，排序返回 Top-K。

**k1 的作用**：控制词频饱和度。k1 越大，词频越高的文档得分增加越显著；k1 越小，单词多次出现的边际收益越低。通常设 1.2~2.0。

**b 的作用**：控制文档长度归一化强度。b=1 时完全按长度归一化，长文档中命中词得分被大幅压缩；b=0 时不做长度惩罚。通常设 0.75，平衡长短文档的竞争公平性。

### 6. RAG 检索如何加速？

1. **384 维轻量向量做粗筛**：粗筛阶段用低维向量（计算量小），减少全量比较开销
2. **HNSW 索引**：图遍历近似检索，避免全量暴力扫描
3. **候选集复用**：第二层 BM25 分数直接复用第一层缓存，不重新检索
4. **并发双路检索**：BM25 和向量检索并发请求，不串行等待
5. **Reranker 仅处理 80 条**：精排输入量控制在可接受范围，整体三级链路约 2s
6. **Kafka 异步解耦**：离线入库走 Kafka，不影响在线检索链路

### 7. 如何衡量检索质量？召回率怎么计算？如何判别召回文档是符合标准的？这个标准谁提供？

**召回率计算**：

$$Recall@K = \frac{\text{前 K 个结果中包含正确文档的数量}}{\text{该问题相关文档总数}}$$

**三个等级**：正确召回（完整命中）、部分召回（命中了相关 Chunk 但不完整）、无召回。

**判别标准来源**：
- 自研评测集（2034 条），由领域专家标注每个问题的正确答案所在文档（Ground Truth）
- 三种提问方式（原文/总结/容错）覆盖不同召回难度
- 对于 Agent 支撑场景，额外评估"Agent 作业上下文获取"——即 Agent 在任务执行中能否精准拿到所需上下文

---

### 8. 在什么时候应该挂 RAG 知识库，什么时候需要进行模型微调？

| 场景 | 选 RAG | 选微调 |
|------|--------|--------|
| 知识时效性强、频繁更新 | ✓（新文档入库即可） | 重新训练成本高 |
| 需要引用来源、可溯源 | ✓（可标注 Chunk 出处） | 模型参数内无法溯源 |
| 领域术语理解不足 | ✗（RAG 只是上下文，模型本身不理解） | ✓（SFT 将术语概念烙印权重） |
| 格式/结构化输出不稳定 | ✗（Prompt 效果有限） | ✓（SFT 将格式肌肉记忆烙印） |
| 需要抑制危险输出倾向 | ✗（护栏是后置拦截，治标） | ✓（DPO 对齐治本） |
| 推理延迟要求极低 | ✗（多了检索环节） | ✓（推理阶段无额外开销） |

实践：两者结合——RAG 解决知识时效性和可溯源性；SFT 解决领域术语理解和格式遵从；DPO 解决危险指令幻觉抑制。三者互补，分别针对不同问题根因。

## 3. 模型微调

### 1. 微调时的数据量级在什么级别？如何构造的？有没有使用模型蒸馏？

**数据量级**：

| 模型                                  | 原始数据量             | 过滤后      | 生成方式                 |     |
| ----------------------------------- | ----------------- | -------- | -------------------- | --- |
| Embedding 微调（gte-multilingual-base） | 75 万 QA 对         | 48 万     | Qwen2.5-72B 自动生成     |     |
| Reranker 微调（gte-reranker）           | 与 Embedding 共用正样本 | 另外构造难负样本 | Embedding Top-50 中采样 |     |
| Query 改写模型（Qwen3-4B SFT）            | 数千条               | —        | 72B 大模型生成 + 人工审核     |     |

**构造方式**：用 Qwen2.5-72B 对知识库文档从三个角度（文档内容/标题/摘要）生成 QA 对，再经 MinHash 去重、Reranker 相关性过滤、指代性问题剔除、长度配比平衡等后处理流水线，最终保留约 64% 的高质量样本。

**模型蒸馏**：Query 改写的 Qwen3-4B 使用了知识蒸馏思路——以 Qwen2.5-72B 生成的高质量改写结果作为 Teacher 输出，对 4B 小模型做 SFT，让小模型学习大模型的指代消解和问题扩展能力，实现"大模型教小模型"的能力迁移。

### 2. 简单介绍一下 LoRA 微调：LoRA 微调时参数初始化用的什么方法？如何评估微调的模型达到了预期效果？

**LoRA 原理**：在原始权重矩阵 $W \in \mathbb{R}^{d \times k}$ 旁并联一个低秩分解矩阵 $\Delta W = BA$，其中 $B \in \mathbb{R}^{d \times r}$，$A \in \mathbb{R}^{r \times k}$，$r \ll \min(d,k)$。前向传播变为 $h = Wx + \frac{\alpha}{r} BAx$，只训练 $A$、$B$ 两个小矩阵，冻结原始权重。推理时将 $\Delta W$ 合并回 $W$，无额外延迟开销。

**参数初始化**：$A$ 用高斯随机初始化（均值 0、小标准差），$B$ 初始化为**全零矩阵**。这样训练开始时 $\Delta W = BA = 0$，模型行为与原始基座完全相同，从一个稳定基线起步，不会引入随机扰动破坏初始性能。

**评估方法**：
- **领域评测集**：自有数据集 Recall@10、NDCG@10 等核心指标是否达到预期提升（如 Reranker NDCG@10 从 0.62 → 0.76）
- **通用基准防遗忘检验**：DuRetrieval / MTEB 等开源集上的指标下降是否在可接受范围（通常容忍 <2pp）
- **领域 Badcase 人工盲测**：专家抽样对比微调前后在典型混淆案例（如 SIB1/SIB2 混淆）上的表现，确认问题确实被修复

### 3. 不同的应用场景是训练不同模型吗？还是说就训练了一个模型呢？

训练了**多个专用小模型**，主干推理仍用通用大模型：

| 模型 | 用途 | 是否微调 |
|------|------|---------|
| Qwen3-32B | 主干推理（用例生成、结果分析） | 否，纯 Prompt |
| gte-multilingual-base 微调版 | 向量化（384维/768维双输出） | 是，领域 SFT |
| gte-reranker 微调版 | 精排（Cross-Encoder 打分） | 是，LoRA |
| Qwen3-4B 微调版 | Query 改写（指代消解 + 问题扩展） | 是，SFT |
| Qwen3-4B 微调版 | 记忆压缩摘要（超出 Token 预算时） | 是，SFT |

**设计理由**：不同任务对模型能力的要求不同——向量化需要语义对齐，改写需要语言理解，两者放在同一模型中会相互干扰且难以独立更新。专用小模型各司其职，延迟低、成本可控，在推理路径上可独立扩容。

### 4. 什么叫拒绝采样？

拒绝采样（Rejection Sampling Fine-tuning，RS-FT）是强化学习对齐中的一种数据筛选策略：

1. 对同一个 Prompt，用当前策略模型采样多个回答（如 N = 16 条）
2. 用 Reward Model（或规则打分器）对每条回答评分
3. 只保留得分最高（或超过分数阈值）的回答作为正样本，**拒绝**其余低质量回答
4. 用筛选后的高质量样本对模型做新一轮 SFT 训练

**作用**：将 RL 探索过程中产生的高质量输出转化为有监督训练数据，比纯 PPO 更稳定、实现更简单。核心思想是"采样→评分→筛优→微调"的迭代循环，DeepSeek-R1、LLaMA 的训练流程中均有此环节。

**与 PPO 的区别**：PPO 是在线 RL，每步更新策略并立即利用新策略采样；拒绝采样是离线过滤，先大量采样再筛选，可以并行化，但无法像 PPO 那样在训练过程中动态调整探索方向。

---

### 5. 讲一下 DeepSeek 训练一个 Step 的流程。

以 DeepSeek-R1 的 GRPO 训练为例，一个训练 Step 的完整流程：

**① 采样阶段（Rollout）**
对一批 Prompt，策略模型（Policy）并行自回归生成 G 个完整回答（Group），通常 G = 8~16。

**② 奖励计算（Reward）**
对每个 (Prompt, Response) 对，规则打分器评分：
- **格式奖励**：思考过程是否在 `<think>...</think>` 标签内
- **答案正确性奖励**：数学题对比最终答案，代码题运行测试用例

**③ 优势估计（Advantage Estimation，GRPO 特有）**
不训练独立的 Critic/Value 网络，而是用组内相对奖励：

$$\hat{A}_i = \frac{r_i - \text{mean}(\{r_j\}_{j=1}^G)}{\text{std}(\{r_j\}_{j=1}^G)}$$

组内得分高于平均值的回答获得正优势，反之获得负优势。

**④ Policy 梯度更新**
用 PPO-clip 目标函数 + KL 散度约束：

$$\mathcal{L} = \mathbb{E}\left[\min\left(\frac{\pi_\theta(a|s)}{\pi_{\text{old}}(a|s)} \hat{A}, \;\text{clip}\!\left(\frac{\pi_\theta}{\pi_{\text{old}}}, 1-\epsilon, 1+\epsilon\right)\hat{A}\right)\right] - \beta\, D_{KL}(\pi_\theta \| \pi_{\text{ref}})$$

KL 项防止策略偏离参考模型（原始 SFT 模型）太远。

**⑤ 梯度同步与权重更新**
多卡（数据并行 + 张量并行 + 流水线并行）All-Reduce 梯度，优化器（AdamW）更新 Policy 权重，完成一个 Step。

### 6. Tokenize 用的分词算法有哪些？BBPE 为什么使用 Unicode 编码？

**常见分词算法**：

| 算法 | 核心思想 | 代表模型 |
|------|---------|---------|
| BPE（Byte Pair Encoding） | 从字符级迭代合并频率最高的相邻 pair | GPT-2 |
| BBPE（Byte-level BPE） | 以字节（byte）而非字符为最小单元做 BPE | GPT-3、LLaMA |
| WordPiece | 类似 BPE，但合并策略是最大化语料似然而非频率 | BERT |
| SentencePiece | 与语言无关，支持 BPE 和 Unigram LM，无需预分词 | T5、LLaMA2、Qwen |
| Unigram Language Model | 从大词表开始剪枝，保留最大化语料似然的最小词表 | AlBERT |

**BBPE 为什么使用字节级（Unicode）编码**：

1. **消除 OOV 问题**：词汇表基础单元是 256 个字节（覆盖所有可能字节值），任何字符序列都能被分解为字节序列，永远不会出现未知词（OOV/`<UNK>`），对罕见语言、emoji、特殊符号完全兼容
2. **词表规模极小**：基础词表只需 256 个 token，BPE 合并后总词表通常在 32K~128K，远小于逐字符词表（中文字符就有数万个）
3. **UTF-8 天然兼容**：UTF-8 将任何 Unicode 字符编码为 1~4 个字节，BBPE 在字节序列上操作，无需对每种语言单独维护字符级词表，一个词表处理全语言
4. **对比普通 BPE 的优势**：普通 BPE 以字符为最小单元，遇到词表外字符（如生僻汉字、小语种字母）直接报错；BBPE 退化到单字节表示，仍能正常处理，不丢失信息

## 4. 模型推理

### 1. Query 进入大模型后的流程是怎样的？怎么判断模型输出是否结束？了解自回归生成吗？

**完整流程**：

1. **Tokenization**：Query 文本 → 分词器（BPE/SentencePiece）→ Token ID 序列
2. **Embedding 查表**：Token ID → Token Embedding（嵌入矩阵中查对应行），叠加位置编码（RoPE / ALiBi）
3. **Transformer Decoder 前向传播**：经过 N 层 Decoder Block，每层：
   - Masked Multi-Head Self-Attention（含 KV Cache 复用历史 K/V 计算结果）
   - FFN（前馈网络，通常是 SwiGLU 激活）
   - RMSNorm + 残差连接
4. **LM Head**：最后一层隐状态 → 线性投影层 → 词汇表大小的 Logits
5. **采样/解码**：对 Logits 做 Temperature 缩放 → Top-p / Top-k 过滤 → 采样得到下一个 Token
6. **自回归循环**：将新 Token 追加到序列末尾，带入步骤 3 重新前向（借助 KV Cache 跳过已算部分），直到输出终止条件

**如何判断输出结束**：
- 模型输出特殊 Token `<EOS>`（End of Sequence），解码器识别后停止循环
- 或达到 `max_new_tokens` 硬截断上限
- 或遇到自定义 `stop` 词列表中的字符串

**自回归生成的本质**：每步只生成一个 token，串行依赖前序所有 token。**KV Cache** 是关键优化：Attention 中每个 token 的 K/V 矩阵在生成后就缓存下来，下一步只需计算新 token 的 Q 与所有历史 K/V 的注意力，避免对整个序列重复计算，将推理复杂度从 $O(n^2)$ 降为 $O(n)$（增量计算）。

### 2. 大模型调用时有哪些参数（比如温度、top_p、惩罚系数等）？

| 参数                          | 作用                                   | 典型值      | 实践建议                          |
| --------------------------- | ------------------------------------ | -------- | ----------------------------- |
| `temperature`               | 控制 Logits 缩放，越低越确定，越高越随机；设为 0 近似贪心解码 | 0.0~2.0  | 事实性生成用 0.0~0.1；创意/改写用 0.7~1.0 |
| `top_p`（核采样）                | 只从累积概率达到 p 的最小候选集中采样，过滤长尾低概率 token   | 0.9~0.95 | 与 temperature 配合，不建议两者都设为极端值  |
| `top_k`                     | 只从概率最高的 k 个 token 中采样                | 20~50    | top_p 和 top_k 通常二选一，top_p 更常用 |
| `max_tokens`                | 最大输出 token 数，防止无限生成                  | 按业务定     | RAG 回答约 512，用例生成约 2048        |
| `frequency_penalty`         | 按已出现频率惩罚 token，出现越多惩罚越重，降低重复率        | 0.0~2.0  | 0.3~0.7 可有效减少啰嗦               |
| `presence_penalty`          | 对任何已出现过的 token 施加固定惩罚（与频率无关），鼓励话题多样性 | 0.0~2.0  | 长文生成时用于增加词汇多样性                |
| `repetition_penalty`（HF 格式） | 将已出现 token 的 logit 除以该系数（>1 降低重复）    | 1.1~1.3  | Transformers 库参数，与上两者不通用      |
| `stop`                      | 遇到指定字符串立即截断，不等 EOS                   | 按业务定     | JSON 输出时可设 `}` 防止模型继续生成       |

**实践**：RAG 系统生成答案时 temperature=0.1 保证事实性；Query 改写时 temperature=0.7 增加问法多样性；结构化 JSON 输出时配合 Function Calling 原生支持比纯靠参数控制更可靠。

### 2. 讲一下项目中 Agent 的工作流程：从用户触发一直到 Agent 返回，会经历哪些组件？每个组件有什么作用？

从用户发起请求到最终返回的完整链路：

1. **FastAPI 接口层**：接收请求，完成鉴权和 SAP 画像解析，为本次会话分配 Thread_ID，触发 LangGraph 图执行，建立 SSE 长连接向前端实时推送进度

2. **意图解析节点**：识别用户意图（信令查询 / 用例生成 / KPI 诊断等），决定后续路由分支，附带置信度分值

3. **Guardrail 节点（前置）**：正则规则库 + 轻量分类器检测是否包含高危指令（如 `format_c`、`reset_all`），命中则直接拦截，不进入后续节点

4. **Query 改写节点**（Qwen3-4B 微调）：指代消解（"上面提到的参数" → 具体参数名）+ 问题扩展（1 个问题 → 1~3 个子查询），增强召回覆盖度

5. **RAG 三级检索节点**：
   - 第一层：ES BM25 + 384维向量并发检索，RSF 融合 → 80 候选
   - 第二层：768维向量精排 → 20 候选
   - 第三层：Reranker Cross-Encoder 精排 → Top 10 Chunk

6. **HITL 节点（可选触发）**：置信度 0.60~0.65 灰度区间、高危场景、或低相似度 Novel Case 时触发。`interrupt_before` 序列化 State 到 Postgres，Celery 推送钉钉通知，专家 Web 端 Review 后调用 resume API 恢复执行

7. **用例生成节点**（Qwen3-32B）：RAG 上下文 + 历史缺陷库 + Few-shot Golden Cases → 生成结构化测试用例 JSON

8. **执行节点**：调用 Simulation_Runner 工具，在 Docker 隔离沙盒中实际运行测试用例，获取真实执行结果

9. **结果判定节点**：统计轨（KPI 假设检验）+ 语义轨（LLM 日志分析）双轨并行判定通过/失败

10. **数据回流**：专家确认的 Golden Case 入 Milvus 长期记忆库，形成数据飞轮

### 3. 项目有没有遇到比较难的问题？怎么解决的？有成就感的地方在哪里？

**最难的工程问题：HITL 线程阻塞**

**问题**：初始实现在 HITL 节点用 `while True` 轮询审批状态。uvicorn worker 数量有限，一旦有多个任务同时等待人工审批，worker 全部被占满，新请求直接 HTTP 503，且 Nginx 30 秒超时触发 504。

**解决过程**：
- 方向一：轮询间隔拉长 + 超时重连，治标不治本，体验差
- 方向二（最终方案）：利用 LangGraph `interrupt_before` 钩子，触发后将完整 State 序列化快照到 Postgres（Checkpoint），子进程立即退出释放 worker；Celery Worker 异步触发钉钉 Webhook 通知专家；专家 Review 后，前端调用唤醒 API，后端按 Thread_ID 从 Postgres 拉取 State，注入修改内容，调用 `.resume()` 恢复执行。

**结果**：等待审批期间零线程占用，支持 100+ 任务并发等待，体验与异步任务队列完全一致。

**最有成就感：Embedding 微调数据工程**

从零设计 75 万 QA 对的生成和后处理流水线。难点在于 LLM 生成质量不稳定，直接训练会导致模型过拟合表面词汇而非语义关系。通过 MinHash 去重 + Reranker 过滤 + 指代性问题剔除 + 长度配比平衡，过滤掉约 36% 低质样本，剩余 48 万条高质量样本完成训练。最终自有数据集 Top10 召回率从 62% 提升到 88%，E2E 准确率达 93%。这个结果证明了**数据质量比数据数量更重要**的判断是正确的。

## 6. 强化学习

### 1. 讲一讲你对 GRPO 的理解。

GRPO（Group Relative Policy Optimization）是 DeepSeek-R1 提出的 RL 算法，是 PPO 的简化变体，核心创新是**用组内相对奖励替代 Critic 网络的价值估计**。

**标准 PPO 的痛点**：需要训练一个独立的 Value（Critic）网络来估计每个状态的期望回报，对于参数量百亿以上的 LLM，维护一个同等规模的 Critic 网络显存开销极大（几乎翻倍）。

**GRPO 的解法**：对同一 Prompt 采样 G 个回答（Group），用组内奖励的相对排名代替 Critic 估计：

$$\hat{A}_i = \frac{r_i - \text{mean}(\{r_j\}_{j=1}^G)}{\text{std}(\{r_j\}_{j=1}^G)}$$

组内得分高于均值的回答获得正优势（值得强化），低于均值的获得负优势（值得抑制）。

**策略更新**（PPO-clip + KL 约束）：

$$\mathcal{L} = \mathbb{E}\!\left[\min\!\left(\frac{\pi_\theta}{\pi_{\text{old}}} \hat{A},\;\text{clip}\!\left(\frac{\pi_\theta}{\pi_{\text{old}}}, 1\!-\!\epsilon, 1\!+\!\epsilon\right)\!\hat{A}\right)\right] - \beta\, D_{KL}(\pi_\theta \| \pi_{\text{ref}})$$

KL 项约束策略不偏离参考模型（原始 SFT 基座）太远，防止"奖励黑客"。

**优势**：去掉 Critic 网络后，显存和计算量约减少 40%~50%，更适合超大规模模型训练。组内对比的思路也让优势估计方差更小、训练信号更稳定。

### 2. DPO 和 PPO 算法有什么区别？为什么 DPO 容易出现模型退化？

**核心区别**：

| 维度 | PPO | DPO |
|------|-----|-----|
| 训练范式 | 在线 RL：采样 → 奖励 → 更新，闭环迭代 | 离线对比学习：直接在静态偏好数据集上优化 |
| Reward Model | 需要单独训练，独立打分器 | 不需要，隐式嵌入损失函数中 |
| 数据 | 可在线自采样，分布随策略更新 | 依赖固定的 (chosen, rejected) 偏好对数据集 |
| 计算成本 | 高（Value 网络 + Reference Model + Policy） | 低（只需 Policy + Reference Model） |
| 稳定性 | 相对复杂，超参敏感 | 收敛快，实现简单 |

**DPO 的损失函数**（隐式奖励对比）：

$$\mathcal{L}_{\text{DPO}} = -\mathbb{E}\!\left[\log\sigma\!\left(\beta \log\frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta\log\frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)}\right)\right]$$

**DPO 容易退化的原因**：

1. **离线数据分布漂移**：偏好数据集是用旧策略采集的，策略更新后与当前模型分布脱节。模型会过拟合静态数据的模式而非学会真正对齐，产生分布外泛化失败。

2. **梯度信号退化（Gradient Collapse）**：损失同时推高 chosen 概率、压低 rejected 概率。若 rejected 在参考模型上概率已极低，该项梯度接近零，模型为了满足损失目标，反而会大幅降低 chosen 的似然（因为两项对比值只要拉大差距即可），导致模型输出分布熵崩塌、生成质量下降。

3. **无独立 Reward Model 约束**：PPO 的 Reward Model 可以对新生成的回答实时打分纠偏；DPO 缺乏这一在线反馈环路，无法判断模型是真正提升了还是在"钻数据集空子"。

**缓解方法**：IPO（Identity Preference Optimization）、SimPO（引入参考无关正则项）、或混合 PPO+DPO 两阶段训练。

### 3. DeepSeek-R1 论文有看过吗？它在模型架构层面、算法层面有什么创新？

**架构层面**（基于 DeepSeek-V3）：

1. **MoE（Mixture of Experts）架构**：FFN 层替换为多个专家网络，每个 token 通过 Top-K 路由只激活部分专家。总参数 671B，每个 token 激活参数约 37B，推理计算量大幅降低，实现"大参数量 + 低推理成本"的平衡。

2. **MLA（Multi-head Latent Attention，多头潜在注意力）**：将 K/V 投影到低维潜空间（Latent Space）再做 Attention，大幅压缩 KV Cache 的显存占用。长文本推理时内存瓶颈显著缓解，相比标准 MHA 节省约 93% 的 KV Cache。

3. **无辅助损失负载均衡**：通过为每个专家维护动态偏置项（Bias）控制路由均衡，而非引入额外辅助 Loss 项。避免辅助 Loss 与主任务梯度相互干扰，负载均衡效果更稳定。

**算法层面**：

1. **GRPO**：去掉 Critic 网络，用组内相对奖励估计优势函数，训练成本减半（见上题）。

2. **纯 RL 冷启动证明（R1-Zero）**：不依赖任何 SFT 监督数据，从基座直接用 RL 训练，证明了推理能力（自我反思、长思考链、回溯）可以完全从强化学习中涌现，不需要人工标注思维链。这是对"CoT 能力来自 SFT 蒸馏"主流观点的有力反驳。

3. **两阶段训练流程**：
   - 阶段一：少量高质量 SFT 冷启动（避免 RL 初期输出格式混乱），数千条专家标注的推理链数据
   - 阶段二：大规模 GRPO 强化训练，模型在规则奖励驱动下自发延长思考深度

4. **规则奖励设计（可验证奖励）**：仅使用两类奖励——格式正确性（思考过程必须在 `<think>` 标签内）和答案正确性（数学题字符串匹配、代码题单元测试通过率）。完全规避了 Reward Model 训练成本和 Reward Hacking 风险，同时保证奖励信号客观无偏。
