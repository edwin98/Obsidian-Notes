# 防幻觉机制全景

> 本文档描述无线豆包 RAG 系统中"答案可信"这一核心原则的完整工程实现。系统首要原则是"不知为不知"，从检索入口到生成出口构建了三层纵深防线。

---

## 总体思路

幻觉的本质是**模型输出的内容与输入的上下文之间存在事实偏差**，可分为两类：

| 类型 | 描述 | 触发条件 |
|------|------|----------|
| **召回型幻觉** | 正确答案根本没进入 LLM 的上下文 | 检索层漏召回 |
| **生成型幻觉** | 上下文里有答案，模型却捏造或混淆 | LLM 指令遵循不足 |

两类幻觉的干预手段完全不同，必须分层治理，不能混为一谈。

---

## 第一层：输入级物理阻断（检索兜底）

**在召回质量不达标时，直接切断对 LLM 的调用，降级为传统搜索。**

### 双重相似度阈值

系统为 BM25 和向量检索分别设置独立阈值：

```
bm25_score_threshold   — BM25 最高得分下限
vector_similarity_threshold — 向量最高余弦相似度下限
```

当两路检索的最高得分**均低于各自阈值**时，系统判定知识库中没有可信内容覆盖该问题，执行 `Return Fast`：

```
召回得分 < 双重阈值
    ↓
放弃 LLM 推理
    ↓
降级为 BM25 关键词高亮搜索结果
    ↓
交回用户人工阅读判断
```

**为什么要物理阻断而非让 LLM 自己判断？**

LLM 本身无法可靠地"知道自己不知道"——当上下文完全不相关时，强指令约束的 Prompt 虽然能在多数情况下触发拒答话术，但边界情况下仍有概率生成看似合理的捏造内容。物理阻断将这一不确定性彻底清零。

### 父文档召回降低上下文割裂

传统固定步长切分会把完整语义割裂到不同的 chunk 中，模型在跨段落推理时容易产生指代混乱。本系统采用**子切片索引 + 父文档召回**策略：

1. 向量检索命中极细粒度子切片（语义纯粹，提高命中率）
2. 通过子切片元数据中的 `parent_id` 指针，向上回溯取出父层级完整章节
3. 将完整章节内容喂给 LLM（避免语义断层）

效果：减少了模型在综合推理时因"只看到了答案的一半"而自行脑补的概率。

## 第二层：推理级强约束（Prompt 工程 + 强制引用）

**假定上下文质量合格，通过精密的 Prompt 设计约束模型只能使用已有证据。**

### 系统提示词（System Prompt）结构

System Prompt 固定负担约 400~650 tokens，包含三个核心约束块：

```
角色定义
└── "你是一个严谨的华为无线通信技术专家"

事实性强约束
└── "请仅基于以下 <context></context> 标签内的参考资料回答问题"
└── "如果不包含相关答案，请输出：'根据当前已知知识库，暂时无法回答该问题'"
└── "严禁捏造不存在的术语、协议编号与事实"

引文格式约束
└── "在每句话结尾必须使用方括号精确标注引用的 doc id"
└── "严禁引用上下文中未提供的 id"
```

### 结构化上下文组装

组装给 LLM 的 Top 10 chunk 必须带有 XML 格式的元数据标记：

```xml
<doc id="doc_1" title="HARQ 参数配置指南 v2.3">
HARQ 最大重传次数由 maxHARQ-Tx 参数控制，取值范围为 1~8 次...
</doc>

<doc id="doc_2" title="无线参数优化手册">
在低速移动场景下，建议将 maxHARQ-Tx 配置为 4 次以平衡...
</doc>
```

doc_id 是 Prompt 约束与前端渲染之间的核心纽带，全链路贯通，不可缺失。

### 强制内联引用（Inline Citation）的防幻觉原理

强制引用不只是前端展示需求，它从根本上改变了模型的生成行为：

**外部化思维链（Externalized Chain-of-Thought）**

强迫模型在输出任何实质观点前，必须在注意力空间中扫描并锚定 `<context>` 块里的文档 ID。这种"生成论点前必须找到落地出处"的机制，让模型做到**找不到依据就不生成**：

```
传统生成：query → 模型内部知识 → 输出
强制引用：query → 扫描 context 找 doc_id → 找到了才输出该论点
```

**生成示例对比**：

```
无约束输出（有幻觉风险）：
  "CA 技术可将峰值速率提升至原来的 N 倍。"

强制引用输出（可溯源）：
  "CA 技术可将峰值速率提升至原来的 N 倍 [doc_3]。"
```

第二种写法迫使模型在生成"N 倍"这个具体数字时，必须能在 doc_3 中找到依据，否则引文锚点本身就会暴露捏造。

### 流式拦截与前端渲染

后端通过 SSE 流式推送，前端在接收过程中实时解析引文标记：

```javascript
// 前端正则实时捕获
const citationRegex = /\[doc_(\d+)\]/g;

// 捕获后结合元数据字典动态渲染为上标超链接
// "...吞吐量提升 20% [doc_1]" → "...吞吐量提升 20% ¹"
// 用户 hover → 气泡弹出摘要
// 用户点击 → 跳转原始系统源文件
```

这一机制带来两个业务价值：
1. **用户侧**：获得独立验证事实的抓手，打破 AI 黑盒不确定性
2. **研发侧**：Bad Case 被点踩时，通过引文**秒级定责**（引了噪音文档 → 召回/重排问题；引对了但结论错 → LLM 生成幻觉）

## 第三层：输出级离线监控（评测闭环）

**假定前两层已经尽力，仍需持续量化残余幻觉并驱动迭代。**

### Faithfulness（事实忠实度）自动评测

每日 E2E 评估流水线中包含专项幻觉检测，采用**逆向声明提取算法**：

```
Step 1：将模型生成的回答拆分为 N 个原子声明（Claims）
        使用 LLM 提取，每条声明是一个独立可核查的事实断言

Step 2：对每条声明，使用 NLI 模型判断是否被 <context> 文本蕴含（Entailment）

Step 3：Faithfulness = 被蕴含的声明数 / N
```

**举例**：

```
模型回答拆分出 5 条声明：
  声明1："HARQ 最大重传次数为 8 次"         → context 中有依据  ✓
  声明2："maxHARQ-Tx 参数取值范围 1~8"      → context 中有依据  ✓
  声明3："重传间隔建议 2ms"                  → context 中无依据  ✗（幻觉）
  声明4："该参数在 RRC 信令中配置"           → context 中有依据  ✓
  声明5："华为设备默认值为 4 次"             → context 中无依据  ✗（幻觉）

Faithfulness = 3/5 = 0.60  （低于阈值 0.85，触发告警）
```

### 三类故障归因体系

系统将不佳回答精确分为三类，避免"模糊归因"导致调优方向错误：

```
故障触发（用户点踩 / Bad Case）
    ↓
自动归因树
    ├── 检查最终 context 的 Top 10 中是否包含答案
    │       └── 不包含 → 【召回失败】
    │                     症状：BM25/向量粗排漏捞
    │                     处理：调整分词词表、同义词库、chunk 步长
    │
    ├── 检查 Top 3000 粗排候选中是否包含答案
    │       └── 包含但未进 Top 10 → 【重排失败】
    │                                症状：Reranker 对领域词汇判断不准
    │                                处理：补充领域数据微调 Cross-Encoder
    │
    └── Top 10 中有答案但模型回答错误 → 【幻觉生成】
                                        症状：指令遵循不足 / temperature 过高
                                        处理：降温（0.1~0.2），缩减上下文规模
```

### 夜间 Auto-Diagnosis 巡检流水线

业务端埋点追踪（点踩、换词重问等 Bad Case）通过 Kafka 消费进入智能诊断池，夜间自动走归因树逻辑，输出统计汇聚报告：

```
Kafka 消费 Bad Case
    ↓
查 Elasticsearch 确认答案源文件是否在 Top 160
    ├── 不在 → 【BM25/向量粗排背锅】
    ├── 在但未进 Top 10 → 【Reranker 背锅】
    └── 在前 10 但模型答错 → 【LLM 幻觉与理解不足背锅】
        ↓
生成周报，指引研发下一周的优先级调优方向
```

## 各层防线的适用范围与局限

| 防线 | 解决的问题 | 无法解决的问题 |
|------|------------|----------------|
| **物理阻断** | 知识库无覆盖时的凭空捏造 | 知识库有内容但内容本身有误 |
| **Prompt 约束 + 强制引用** | 模型脱离 context 自由发挥 | 模型误读 context 中的内容 |
| **离线 Faithfulness 评测** | 量化残余幻觉，定位问题 | 实时拦截（事后检测） |
| **夜间归因巡检** | 自动定责、驱动迭代 | 无法替代人工对复杂 Bad Case 的深度分析 |

三层防线协同工作，在实测中将端到端准确率维持在 **93%**。生成型幻觉（Faithfulness < 0.85 的比例）在硬域（5G 技术问答）下低于 **4%**。

## 关键设计决策

**1. 为什么不依赖 LLM 的内置拒答能力？**

LLM 内置的"我不知道"是概率性行为，在边界情况下不稳定。物理阻断是确定性的，零误差。

**2. 为什么强制引用比单纯 Prompt 约束更有效？**

单纯说"不要捏造"是负向约束，模型可能在 temperature 高或 context 长时遗忘。强制引用是正向的行为设计——每个论点都需要一个锚点，锚点的存在本身就是对内容忠实性的隐式核查。

**3. 为什么选 72B 而非小模型？**

System Prompt 包含极为复杂的行为约束（禁止捏造、强制 inline citation、严格角色扮演）。72B 在指令遵循的鲁棒性上显著强于 32B，极少出现"遗忘约束"或"引用了不存在的 doc_id"的边界情况。对于"答错代价极高"的企业研发场景，这 8~12% 的质量增益有直接业务价值。
