# Note Polisher Workflow

当你被要求运行 `note-polisher` 工作流时，请严格按照以下步骤对目标 Markdown 笔记进行优化、润色和内容补全，特别适用于技术笔记的整理。

## 执行步骤

1. **阅读与语境分析**
   - 完整读取目标 Markdown 文件的内容，理解笔记的核心主题、上下文结构以及用户想要表达的重点。

2. **排版与格式规范化 (Formatting)**
   - **标题层级梳理**：修复混乱的标题层级（例如，确保 `##` 正确嵌套在 `#` 之下，避免同级堆砌）。
   - **中英混排优化**：强制在英文单词、数字与中文字符之间增加一个半角空格，提升阅读体验（例如：`使用 Python 编写脚本` 而不是 `使用Python编写脚本`）。
   - **块级元素隔离**：确保所有的代码块（` ``` `）和独立公式块（`$$`）前后各保留一个空行，以保证 Obsidian 等渲染器正确解析。
   - **将长段落列表化**：识别出具备并列、递进关系的冗长段落，将其转化为无序 (`-`) 或有序 (`1.`) 列表。

3. **措辞润色与表达提升 (Wording Polish)**
   - **专业化表达**：将口语化、生硬或晦涩的翻译腔转化为清晰、准确的技术书面语。
   - **信息高亮提取**：自动识别段落中的特定专有名词、核心概念和重要结论，并使用 `**加粗**` 的形式进行强调，让读者能快速抓住重点。
   - **精简冗余**：剔除冗余的连接词和没有实质意义的“废话”，做到言简意赅。

4. **技术逻辑校对与细节延展 (Correctness & Detail Expander)**
   - **事实与逻辑检查**：如果发现技术常识可能存在错误（如公式写错、概念混淆），或者上下文出现逻辑断层，请不要直接大砍大改，而是使用 Obsidian 的 Callout 语法给出提醒，例如：
     ```markdown
     > [!warning] 逻辑检查
     > 这里关于“卷积核”的描述似乎与前文“Transformer”的主题存在冲突，建议确认。
     ```
   - **概念扩展与补充**：如果上下文中出现了未加解释的技术缩写（如 `ViT`, `RoPE` 等）或抽象概念，请在合适的位置或以引用的形式 `> [!info] 补充说明` 补充其关键定义。
   - **增加示例/对比表**：当发现文章只有干瘪的理论时，主动尝试补充一个极简的代码片段或应用场景的对比表格，使概念更具象。

5. **执行修改与反馈**
   - 使用文件编辑工具（如 `replace_file_content` 或 `write_to_file`）将优化后的内容写回文件。
   - 保持用户原始观点的初衷，**绝不随意篡改用户原本正确的意思**。
   - 更新完成后，向用户提供一份简短的“修改报告”，列出你主要做了哪些方面的优化。
