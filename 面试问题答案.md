## 1. chunk 的时候，表格的处理方式，不同版本文档的处理方式

### 表格处理

表格根据来源格式分两种路径处理：

**结构化表格（Excel/CSV）**：直接转换为纯文本（每行一条记录，列名作为字段前缀），然后按通用 chunk 策略切分。表格不会在行中间被截断——若表格超过 chunk 上限，会以逻辑行为单位分片，并在每个分片中保留表头，保证每个 chunk 可以独立理解。

**文档内嵌表格（Word/PDF/HTML 中的 Markdown 表格）**：统一先转为 Markdown 格式，然后整张表格作为一个不可拆分的叶子节点处理。如果表格本身超出 512~800 Token 的叶子 chunk 上限，则由 LLM 生成摘要替代（同非叶子节点策略），保留表格结构含义，同时控制 chunk 体积。

**表格的元数据处理**：切片时会注入父级章节标题到 metadata，确保表格 chunk 在被召回后，LLM 能通过上下文理解该表格属于哪个功能点或规范章节。

### 不同版本文档处理

版本管理的核心原则是**同一份文档只保留一条物理记录**，避免多版本同时存在导致召回时出现矛盾答案：

- **设计文档**：仅保留最新版本入库，旧版本在更新时触发离线 Pipeline 标记删除并重新入库，同时在 metadata 中注入 `pdu_tags` 和版本号信息。
- **3GPP 协议**：不同协议编号对应不同物理记录（R15/R16/R17 均保留），通过文件名与协议编号的重映射关系统一标准化，查询时可以通过标量过滤限定版本范围。
- **版本冲突降噪**：入库时以文档的唯一 ID（如文件路径或源系统 ID）作为主键，重新入库时用 Upsert 覆盖而不是追加，从根本上保证同一文件只有一个向量表示存在于库中。

## 2. RAG 系统为了提高效率做了哪些工作

效率优化分在线链路和离线链路两个维度：

### 在线链路（降低 P99 延迟）

**并发检索**：BM25（ES）和 384 维向量检索并发发起，不串行等待，第一层 ~0.5s 内完成。

**两级向量粒度分工**：第一层用轻量 384 维做大范围粗筛（百万级），第二层才用 768 维做精语义重算，避免对全量文档做高维计算。

**BM25 分数缓存复用**：第二层精筛中，BM25 分数直接复用第一层缓存结果，不对全量知识库重新检索，减少一次全量 BM25 开销。

**小模型承担轻任务**：Query 改写（指代消解 + 问题扩展）用 Qwen3-4B 而非大模型，单次改写耗时控制在 1s 内；上下文摘要压缩也走 4B 模型异步 Celery 任务，不阻塞主链路。

**SSE 流式推流**：答案生成走 SSE 逐 token 流式推送，用户感知首字节时间大幅优于等待完整答案，降低主观等待感。

### 离线链路（提升入库吞吐）

**Kafka 异步解耦**：在线问答链路和离线文档入库链路被 Kafka 消息总线完全隔离，大批量文档入库（PDF/Word/Excel）不会冲击在线服务稳定性。

**GPU/CPU 异构 Worker 分流**：图片、PPT 等计算密集型走 GPU Worker，通用文本处理走 CPU Worker，按任务特性分配资源，提升整体吞吐。

**Celery 异步任务**：上下文轮次溢出监控、历史摘要压缩等非实时任务全部异步，不占用请求处理线程。

## 4. InfoNCE Loss 的原理

InfoNCE（Info Noise Contrastive Estimation）是对比学习的核心损失函数，本质是一个分类任务的交叉熵——让模型从 K 个负样本中识别出正确的正样本。

公式：

$$\mathcal{L} = -\log \frac{\exp(\text{sim}(q, d^+) / \tau)}{\exp(\text{sim}(q, d^+) / \tau) + \sum_{i=1}^{K} \exp(\text{sim}(q, d_i^-) / \tau)}$$

**直觉理解**：分子是 query 和正样本文档的相似度经温度缩放后的指数；分母是正样本加上所有负样本的同类指数之和。损失最小化意味着正样本相似度要尽可能大于所有负样本——等同于让模型把"从 K+1 个候选中找到正确文档"这道题答对。

**温度系数 τ 的作用**：τ 越小，相似度差异被放大越剧烈，梯度越集中在难分的边界样本上，但也越容易训练不稳定。实际取值从 0.05 开始，随训练逐步降到 0.01。

**In-batch Negatives**：实现时同一 batch 内其他样本的正文档自动作为当前样本的负样本，batch_size=1024 时每个样本有 1023 个 in-batch 负样本，无需额外构造大量负例，计算高效。

**为什么 RAG 用这个 Loss**：RAG Embedding 微调的目标是"查询-文档相关性"而不是"语义相似度"。InfoNCE 天然对应检索场景——"从大量文档中找到与 query 最相关的那个"，与召回任务目标完全对齐。

## 5. Embedding 微调和 Reranker 微调用的什么框架

**Embedding 微调**：

基于 HuggingFace `transformers` + `sentence-transformers` 训练框架，结合 `GradCache` 库实现大有效 batch（单卡 A30 24GB，batch_size=64，GradCache 分块=16，双卡 DDP 后有效 batch 达 1024）。

损失函数为 InfoNCE（自定义实现），第三阶段叠加 Cross-Encoder 蒸馏（KL 散度损失），联合优化。

基座模型：`gte-multilingual-base`（305M，阿里 GTE 系列，HuggingFace 上直接加载）。

训练环境：A30 × 2，CUDA 12.1，FP16，PyTorch DDP 数据并行。

**Reranker 微调**：

同样基于 HuggingFace `transformers`，Cross-Encoder 架构，输入格式为 `[CLS] query [SEP] document [SEP]`，从 `[CLS]` 位置接全连接层输出相关性分数。

损失函数为 Binary Cross-Entropy（Pointwise）或 Listwise Softmax（Listwise 格式，每个 query 配 1 正 + 4~6 负构成一个训练实例）。

基座模型：`gte-multilingual-reranker-base`（305M），选型依据是 305M 在 A10G 上精排 50 个候选文档的 P99 延迟可以控制在 200ms 内。

## 6. Reranker 的工作方式，内部是什么结构？和 Embedding 模型的区别？

### Reranker 工作方式

Reranker 采用 **Cross-Encoder** 架构。工作时将 query 和 document **拼接**成单一序列送入 Transformer：

```
输入：[CLS] query [SEP] document [SEP]
输出：从 [CLS] 位置取 hidden state，经全连接层输出 1 个相关性分数
```

因为 query 和 document 的 token 在同一个序列里做**全局注意力**，每个 query token 可以直接 attend 到 document 中的每个 token，反之亦然，能捕捉细粒度语义关系——比如否定词、数字精确匹配、条件依赖关系等。

**例子**：对于 query "HARQ 最大重传次数如何配置"，Bi-Encoder 对"HARQ 重传机制"和"HARQ 最大重传次数"两篇文档打分接近；Cross-Encoder 会让"最大"和"次数"这两个 token 直接 attend 到文档中对应的参数名和数字，区分度显著。

### 与 Embedding（Bi-Encoder）的区别

| 维度 | Bi-Encoder（Embedding） | Cross-Encoder（Reranker） |
|------|------------------------|--------------------------|
| 编码方式 | query 和 document 独立编码为向量 | query + document 拼接后联合编码 |
| 交互粒度 | 只有最终向量的余弦相似度 | token 级全局注意力交互 |
| 是否可预计算 | document 向量可离线预计算 | 无法预计算，每次必须重新计算 |
| 检索复杂度 | O(1) ANN 检索（向量库） | O(N) 逐对计算 |
| 精度 | 较低（缺少细粒度交互） | 较高（完整 token 级交互） |
| 适用场景 | 大规模召回（百万级候选） | 小范围精排（50~200 候选） |

**系统中的分工**：Bi-Encoder 负责第一、二层召回（从千万级缩小到 80 个候选），Reranker 负责第三层精排（从 80 个选出最终 Top 10），两者串联互补，兼顾效率和精度。

## 7. Agent 的正常工作流程是什么

系统采用基于 LangGraph 状态机的 **ReAct 循环**架构，核心流转如下：

```
用户请求（自然语言/YAML 测试需求）
     ↓
FastAPI 接口层 → LangGraph 状态机引擎
     ↓
[Agent 节点] LLM 推理（Thought）
  - 解析意图，决策下一步需要调用哪个工具及参数
  - 若无需工具，直接进入结果判定
     ↓
[条件路由] should_continue()
  - 有工具调用 → [Guardrail 节点]
  - 无工具调用 → [结果判定节点]
  - 连续错误超限 / 置信度过低 → [HITL 节点]
     ↓
[Guardrail 节点] 安全检测
  - 对工具调用参数做关键词检查 + 轻量分类器检测
  - 无高危操作 → [Tool 执行节点]
  - 检测到高危操作 → [HITL 节点]
     ↓
[Tool 执行节点] 执行工具（Action）
  - 返回 Observation，追加至 State 的 messages 列表
  - 回到 Agent 节点继续下一轮推理
     ↓
[结果判定节点] 双轨并行判定
  - 统计轨：KPI 均值/方差、T-Test/KS-Test
  - 语义轨：LLM 解析信令日志，根因推理
  - 输出置信度分数
     ↓
置信度 >= 0.65 → 自动输出结论（END）
置信度 < 0.65  → [HITL 节点] 人工介入
     ↓
[HITL 节点] 人工审批
  - State 快照序列化至 Postgres，释放计算资源
  - 触发钉钉告警 Webhook，专家在 Web 端 Review
  - 专家修改参数 → Approve → 从 Postgres 恢复 State，.resume() 继续执行
```

**防死循环保障**：硬熔断（LangGraph `recursion_limit` = 15 跳）+ 软熔断（连续工具错误 ≥ 3 次或置信度 < 0.65 路由至 HITL）双重机制。

## 8. 文档的量级

**知识库规模**：

- 原始企业文档涵盖 PDF/Word/Excel/Web/Wiki/iCase 等多种格式，来源包括 Support 文档、DTS 工单、3GPP 协议、华为知道/华为案例等
- 离线切分后知识库共约 **19 万个 Chunk**（叶子节点 + 非叶子摘要节点）
- Milvus 向量库存储：384 维 + 768 维双粒度，HNSW 索引，支持标量过滤（CQIC 方向、PDU 分类等）
- BM25 词频库：单一统一知识库，挂载 6800 个无线领域专有词

**训练数据量级**：

- Qwen2.5-72B 合成 QA 对：75 万条原始语料，经清洗后 **48 万条**参与 Embedding/Reranker 微调
- 每个 Chunk 平均生成 4 个问题（19 万 Chunk × 4 ≈ 76 万）

**在线检索量级（三级漏斗）**：

- 第一层粗筛：ES+向量各召回 1500，合并去重约 **3000 条候选**
- 第二层精筛：RSF 融合打分取 **Top 80**
- 第三层精排：Reranker 输出最终 **Top 10**

## 9. Agent 调用工具的时候，工具的设计和输入输出参数

### 工具的设计原则

工具通过 **Function Calling / Tool Use** 机制暴露给 LLM。每个工具的定义包含三部分，本质是一段告诉 LLM"这个工具能做什么、怎么调用"的 JSON Schema：

```python
{
    "name": "Test_Case_Query",
    "description": "根据测试场景、频段、特性从知识库检索相关测试用例。当需要获取某个特性的测试点或用例列表时调用此工具。",
    "parameters": {
        "type": "object",
        "properties": {
            "scenario": {
                "type": "string",
                "description": "测试场景名称，如 '切换场景'、'随机接入场景'"
            },
            "feature": {
                "type": "string",
                "description": "被测特性，如 'CA'、'HARQ'、'PRACH'"
            },
            "band": {
                "type": "string",
                "description": "频段标识，如 'n78'、'n257'，不确定时传 null"
            },
            "top_k": {
                "type": "integer",
                "description": "返回用例数量上限，默认 20",
                "default": 20
            }
        },
        "required": ["scenario", "feature"]
    }
}
```

**Description 是最关键的 Prompt**：LLM 不看工具实现代码，仅凭 `description` 和参数描述决定是否调用、传什么参数。描述要精确到"什么情况下用这个工具、不用哪个工具"，避免 LLM 调用错工具。

### 系统中核心工具的输入输出

| 工具 | 关键输入参数 | 输出格式 |
|------|------------|---------|
| `Test_Case_Query` | `scenario`, `feature`, `band`, `top_k` | JSON 数组，含用例 ID、描述、优先级 |
| `Simulation_Runner` | `case_ids: List[str]`, `env_config: dict` | `{"session_id": "xxx", "status": "running"}` |
| `Metrics_Collector` | `session_id: str`, `kpi_names: List[str]` | `{"kpi_name": {"mean": x, "std": y, "samples": [...]}}` |
| `Baseline_Comparator` | `current_metrics: dict`, `baseline_version: str` | `{"result": "PASS/FAIL", "p_value": 0.03, "details": {...}}` |
| `Log_Analyzer` | `session_id: str`, `log_type: "RRC/NAS/PDCP"` | `{"anomalies": [...], "root_cause": "..."}` |

### 输出处理

工具执行结果作为 `ToolMessage` 追加到对话历史（State 中的 `messages`），LLM 在下一轮推理时以此为 Observation 继续决策。输出必须是结构化 JSON——LLM 处理纯文本日志效果差，而且难以做置信度计算。

**错误处理**：工具执行若抛异常，将错误信息作为 ToolMessage 返回给 LLM（不直接崩溃），LLM 看到报错后通常能自动修正参数并重试；连续错误 ≥ 3 次触发软熔断，路由至 HITL 人工介入。

## 10. Agent 系统中测试用例的典型例子，以及测试用例是如何存储的

### 典型测试用例举例

系统面向 5G 无线网络测试，测试用例覆盖三大类场景：

**切换场景（Handover）**

```
用例 ID：TC_HO_001
名称：同频切换成功率验证
场景：UE 在两个 gNB 之间执行基于 A3 事件触发的 X2 切换
前置条件：UE 驻留源小区，RSRP 满足 A3 触发阈值（offset=3dB, TTT=160ms）
执行步骤：
  1. 触发 RRC 测量上报（MeasurementReport）
  2. 源 gNB 下发 RRCReconfiguration（含切换命令）
  3. UE 向目标小区发起随机接入（PRACH）
  4. 完成 RRC 重配置，路径切换（PathSwitchRequest）
预期结果：切换成功率 >= 98%，中断时间 < 50ms
关联协议：3GPP TS 38.331 Section 5.4.3
```

**随机接入场景（PRACH）**

```
用例 ID：TC_PRACH_003
名称：高负载下竞争随机接入成功率
场景：50 个 UE 同时发起竞争随机接入（Contention-Based RA）
前置条件：小区负载 > 70%，PRACH 配置 Format 0，preambleTransMax=10
执行步骤：
  1. 50 个 UE 并发发送 Preamble（Msg1）
  2. 基站响应 RAR（Msg2），含 TA 和上行授权
  3. UE 发送 RRC Setup Request（Msg3）
  4. 基站完成竞争解决（Msg4）
预期结果：随机接入成功率 >= 95%，平均接入时延 < 20ms
关联协议：3GPP TS 38.321 Section 5.1
```

**CA（载波聚合）场景**

```
用例 ID：TC_CA_007
名称：双 CC 下行聚合吞吐量验证
场景：UE 配置 2CC CA（PCC: n78 100MHz + SCC: n41 60MHz）
前置条件：两路 CC 信号质量均 >= -85dBm，SCC 已激活
执行步骤：
  1. 下发 RRCReconfiguration 激活 SCC
  2. 触发全缓冲 FTP 下载（iperf3）
  3. 持续 60s 采集 PDCP 层吞吐量
预期结果：峰值下行吞吐量 >= 2.4Gbps，SCC 激活时延 < 100ms
关联协议：3GPP TS 38.331 Section 5.7.1
```

### 测试用例如何存储

测试用例在系统中以**双索引结构**存储，同时支持关键词精确检索和语义检索：

**Milvus 向量库（语义检索）**

每条测试用例拆分为两级存储：

```
用例描述（叶子 Chunk，512~800 Token）→ Embedding → 384/768 维向量
用例摘要（父级节点，含场景标签）      → Embedding → 向量

metadata 字段：
  - case_id: "TC_HO_001"
  - scenario: "切换"
  - feature: "X2切换"
  - band: "n78"
  - priority: "P1"
  - spec_ref: "38.331-5.4.3"
  - version: "R16"
  - is_golden: true/false（是否经专家确认的 Golden Case）
```

metadata 中的标量字段支持过滤查询（如只检索 P1 优先级或 Golden Case）。

**Elasticsearch（BM25 关键词检索）**

用例中大量 3GPP 术语（PRACH、HARQ、RRC、PDCP 等）依赖无线专有词表分词，BM25 负责精确匹配协议编号和参数名（如 `preambleTransMax=10` 这类数字配置项语义检索会漏召回）。

**Agent 查询时的流转**

```
Agent 推理：需要查切换相关用例
     ↓
调用 Test_Case_Query(scenario="切换", feature="X2切换", band="n78")
     ↓
Tool 内部走三级召回（ES + Milvus 粗筛 → RSF 融合 → Reranker 精排）
     ↓
返回 Top-K 用例列表（含 case_id、步骤摘要、关联协议）
     ↓
Agent 将用例内容注入 Prompt，决策是否触发 Simulation_Runner 执行
```

**Golden Case 的特殊地位**

经 HITL 专家审核通过的用例会被标记 `is_golden=true`，在检索时作为加权正样本优先召回。同时这批数据会持续回流进入 Embedding/Reranker 的微调训练集，形成数据飞轮——系统使用越久，召回的用例质量越高。

## 11. Agent 工具调用全流程详解

### 一、工具的定义

工具（Tool）的本质是一段告诉 LLM "这个功能存在、能做什么、怎么调用" 的 **JSON Schema 声明**。模型不会看也不能执行工具背后的代码，它只能读 Schema。

一个完整的工具定义包含三部分：

```python
{
    "type": "function",
    "function": {
        # 1. 工具名称（模型调用时引用的标识符）
        "name": "Metrics_Collector",

        # 2. 功能描述（最关键，模型选工具的唯一依据）
        #    需要精确描述：什么情况下用这个工具、不用哪个工具
        "description": (
            "通过 SSH/RPC 从网元实时拉取指定会话的 KPI 日志数据。"
            "当测试已通过 Simulation_Runner 启动（拿到 session_id 后），"
            "需要采集实际执行结果时调用此工具。"
            "不要在测试尚未启动时调用此工具。"
        ),

        # 3. 参数 Schema（告诉模型每个参数是什么、类型、是否必填）
        "parameters": {
            "type": "object",
            "properties": {
                "session_id": {
                    "type": "string",
                    "description": "由 Simulation_Runner 返回的测试会话 ID"
                },
                "kpi_names": {
                    "type": "array",
                    "items": {"type": "string"},
                    "description": (
                        "需要采集的 KPI 指标名称列表，"
                        "如 ['throughput_dl', 'ho_success_rate', 'prach_success_rate']"
                    )
                },
                "duration_sec": {
                    "type": "integer",
                    "description": "采集时长（秒），默认 60",
                    "default": 60
                }
            },
            "required": ["session_id", "kpi_names"]
        }
    }
}
```

**工具描述的编写原则**：
- 告诉模型"什么时候该用"，同样重要的是"什么时候不该用"（防止误调用）
- 参数描述要给出合法值的例子，不要只写 "字符串类型"
- 工具数量不宜过多（超过 20 个时模型选择准确率明显下降），可用 Router 或 RAG 动态注入当前上下文相关的工具子集

---

### 二、内部原理：模型是怎么学会调用工具的

**Function Calling 是 SFT 训练出来的能力，不是 Prompt 技巧。**

在模型预训练阶段，LLM 只会输出普通文本。Function Calling 能力来自后续的监督微调（SFT）——在训练数据中加入大量"工具调用对话"样本，让模型学会在特定条件下输出结构化的工具调用 token，而不是普通文本。

模型输出工具调用时，实际上是在生成一段特殊格式的 token 序列（各厂商实现略有差异，但结构类似）：

```
# 模型内部生成的原始 token（以 OpenAI 格式为例）
finish_reason: "tool_calls"
tool_calls: [
  {
    "id": "call_abc123",
    "type": "function",
    "function": {
      "name": "Metrics_Collector",
      "arguments": "{\"session_id\": \"sim_20240315_001\", \"kpi_names\": [\"throughput_dl\", \"ho_success_rate\"], \"duration_sec\": 60}"
    }
  }
]
```

注意 `arguments` 是 **JSON 字符串**而不是对象，需要在代码层 `json.loads()` 解析。`finish_reason` 为 `"tool_calls"` 是区分"模型要调用工具"和"模型直接输出答案"的关键标志。

**模型如何决定调用哪个工具**：模型在 Attention 机制中同时看到了对话历史 + 所有工具的 Schema，其决策过程等同于一个分类+生成任务：先判断当前 Thought 需要什么信息，再从注册的工具列表中选择最匹配描述的工具，最后生成符合该工具 Schema 约束的参数 JSON。

---

### 三、工具调用的完整四步流转

```
Step 1：注册工具 + 发送请求
┌────────────────────────────────┐
│ 开发者代码                      │
│  工具 Schema 列表               │
│  + 用户消息（对话历史）          │  ──► LLM API（携带 tools 参数）
└────────────────────────────────┘

Step 2：LLM 决定暂停并请求工具
                                    LLM 返回 ──►  finish_reason = "tool_calls"
                                                  tool_calls[0].function.name = "Metrics_Collector"
                                                  tool_calls[0].function.arguments = "{...}"

Step 3：开发者代码执行工具（本地/微服务）
┌────────────────────────────────┐
│  解析 tool_calls               │
│  路由到对应函数                 │
│  执行真实业务逻辑               │  ──► 返回真实结果
└────────────────────────────────┘

Step 4：将工具结果追加到消息历史，再次调用 LLM
[
  {"role": "user",      "content": "帮我分析这次切换测试的结果"},
  {"role": "assistant", "content": null, "tool_calls": [...]},  ← Step 2 的输出
  {"role": "tool",      "tool_call_id": "call_abc123",          ← Step 3 的结果
                        "content": "{\"throughput_dl\": {\"mean\": 2341.5, ...}}"}
]
          ──► 再次请求 LLM，模型看到 Observation 后生成最终回答
```

**关键细节**：Step 2 的模型输出（tool_calls）和 Step 3 的工具结果，必须都追加到消息历史中一起发给模型，否则模型在 Step 4 不知道自己之前做了什么。这就是 LangGraph 中 `add_messages` 注解的作用——自动维护这条完整的消息链。

---

### 四、工具的返回内容

工具返回内容统一封装为 `ToolMessage`，其 `content` 字段应为结构化 JSON 字符串（不要返回纯文本日志）：

**好的返回格式**：
```json
{
    "status": "success",
    "session_id": "sim_20240315_001",
    "duration_sec": 60,
    "metrics": {
        "throughput_dl": {
            "mean": 2341.5,
            "std": 87.3,
            "p95": 2489.2,
            "unit": "Mbps",
            "samples_count": 60
        },
        "ho_success_rate": {
            "mean": 0.987,
            "total_attempts": 152,
            "total_success": 150
        }
    }
}
```

**差的返回格式**（LLM 难以解析，且不利于置信度计算）：
```
"切换成功率为98.7%，下行吞吐量均值2341.5Mbps，采集了60秒..."
```

返回结构化 JSON 的好处：
1. LLM 解析准确率更高（不需要从自然语言中抽取数字）
2. 后续 `Baseline_Comparator` 等工具可以直接消费这个 JSON
3. 结果判定节点可以直接对 `mean/std` 做统计检验，不依赖 LLM 转述

## 12. Embedding/Reranker 微调走过的弯路，如果重做要怎么做

### 走过的弯路

**弯路 1：负样本质量差**

初期负样本全用随机采样（从语料库随机取文档），太容易区分，模型很快就能把正负样本分开，但对真实检索没有提升。效果验证时 Recall@10 比预期低 15 个点才发现问题。

**弯路 2：先训练后建评测集**

先跑完训练，再去构建评测集，结果评测集和训练集存在数据泄漏（同一个文档切出来的 QA 对分到了训练集和测试集）。最终 Recall 数字好看，但真实 A/B 效果不理想。

**弯路 3：Reranker 输入长度没有控制**

Reranker 输入格式是 `[CLS] query [SEP] document [SEP]`，早期没有对 document 做截断，超过 512 token 的文档被直接截断，关键信息（往往在文档末尾）丢失，精排效果反而不如 Bi-Encoder。

**弯路 4：Listwise 训练格式引入太晚**

Reranker 最初用 Pointwise（Binary BCE，每个 query-doc 对独立打分），效果一般。后来换 Listwise（每个 query 配 1 正 + 5 负构成一个训练实例，Softmax 联合优化），Recall 提升明显，但切换成本较高。

### 如果重做

1. **先建评测集，再做任何训练**：按文档粒度切分 train/val/test，防止泄漏；把评测集当作不可动的"真相"，所有迭代决策都以它为准。
2. **Hard Negatives 从第一轮开始**：用 BM25 召回的 Top 50 去掉真正相关文档，剩下的作为 hard negatives；再加一轮用训练中途 checkpoint 挖 false negatives（被模型误判为正样本的负例）。
3. **Reranker 直接用 Listwise**：不走 Pointwise 这条弯路，训练格式从一开始就是 1 正 + 5 负的 Listwise Softmax。
4. **输入长度从一开始就控制**：document 最多保留 256~400 token（覆盖核心内容），query 留 64~128 token，总长控制在 512 token 以内。
5. **阶段式训练**：第一阶段用大量弱监督 QA 对预热，第二阶段再加 hard negatives 精调，第三阶段叠加 Cross-Encoder 蒸馏（用 Reranker 的软标签蒸馏 Bi-Encoder）。

## 13. 模型在 Agent 中具体负责什么，是否参与用例的生成

### LLM 在 Agent 中的核心职责

LLM 承担 Agent 的"大脑"角色，具体负责：

1. **意图理解**：解析用户的自然语言请求，识别需要验证的测试场景、频段、特性等关键信息。
2. **工具选择与参数生成**（Function Calling）：根据当前 Thought，从注册的工具列表中选择最合适的工具，并生成符合 Schema 约束的参数 JSON。
3. **Observation 解读**：看到工具返回结果后，判断是否达到目标，还是需要继续调用下一个工具。
4. **根因推理**：在结果判定节点，LLM 解读信令日志中的异常（如 HARQ 重传率过高），结合协议知识给出根因分析文本。
5. **最终报告生成**：综合所有 KPI 和对比结果，生成结构化的测试结论供人工审阅。

### LLM 是否参与测试用例的生成

**离线阶段（参与度高）**：

使用 Qwen2.5-72B 从 3GPP 协议文档和设计文档中大批量合成 QA 对，这些 QA 对作为 Embedding/Reranker 微调的训练数据（48 万条）。合成逻辑是：给 LLM 一段文档 chunk，让它生成"什么问题会检索到这段内容"，以此构建 (query, positive_doc) 对。

**在线阶段（参与度低，有约束）**：

在线阶段 LLM 不会直接"凭空"生成新用例，而是：
- 通过 RAG 检索已有的用例作为参考
- 基于现有用例 + 协议约束生成参数变体（如同一场景换频段）
- 所有 LLM 生成的用例变体必须经 HITL 专家审核（`is_golden=false`），审核通过后才能标记为可信用例，回流训练集

核心原则：LLM 生成的内容不直接入库，必须经过人工或统计验证。

## 14. RAG/Agent 评测中，哪些评测指标有 LLM 的参与

### RAG 系统评测

**纯统计指标（无 LLM）**：
- **召回指标**：Recall@K、MRR（Mean Reciprocal Rank）、NDCG——只需要标注的相关文档集合，不需要 LLM
- **BM25 分数**：词频统计，纯计算

**有 LLM 参与的指标**：
- **答案相关性（Answer Relevance）**：用 LLM 判断生成答案是否与原始问题相关（用于无参考答案的场景）
- **忠实度（Faithfulness）**：用 LLM 检查答案中的每个声明是否能在检索到的上下文中找到支撑，识别幻觉
- **上下文精度（Context Precision）**：用 LLM 判断检索到的文档中，哪些对生成答案真正有用（区分噪声文档）
- **端到端 QA 评分**：没有 Golden Answer 时，用 LLM-as-Judge 打分（通常用 GPT-4 对比两个系统的输出）

### Agent 系统评测

**纯统计**：
- 工具调用次数、平均轮次数、置信度分布
- KPI 统计检验（T-Test、KS-Test）——完全是数学，无 LLM

**有 LLM 参与**：
- **任务完成度（Pass Rate）**：判断 Agent 最终输出是否"正确完成了任务"，往往需要 LLM 理解语义（特别是边界情况）
- **根因分析准确性**：LLM 生成的根因文本与专家标注做语义相似度对比，需要嵌入模型计算
- **工具调用合理性**：用 LLM 判断 Agent 的工具选择路径是否合理（反事实分析）

实际工程中，我们尽量把 LLM 评测限制在"非统计能覆盖的部分"，减少 LLM 评测的成本和随机性。能用 Recall@K 衡量的就不用 LLM-as-Judge。

```python
            "function": {
                "name": "Test_Case_Query",
                "arguments": "{\"scenario\": \"切换\", \"feature\": \"X2切换\", \"band\": \"n78\"}"
            }
        }
    ]
}

# 形态 2：直接输出答案（finish_reason = "stop"）
{
    "role": "assistant",
    "content": "n78 频段 X2 切换测试整体判定：PASS（置信度 91%）...",
    "tool_calls": null
}
```

### ToolMessage（工具执行结果）
```python
{
    "role": "tool",
    "tool_call_id": "call_abc123",   # 与 AIMessage 中的 tool_calls[i].id 对应
    "name": "Test_Case_Query",        # 可选，部分实现会带工具名
    "content": "{\"status\": \"success\", \"cases\": [...]}"   # 必须是字符串，JSON 序列化后的结果
}
```

### 关键约束
- `tool_calls` 中的 `arguments` 是 **JSON 字符串**，不是对象，代码层需要 `json.loads()` 解析
- ToolMessage 的 `tool_call_id` 必须与对应 AIMessage 中的 `id` 完全匹配，否则 LLM 无法关联上下文
- AIMessage（含 tool_calls）和 ToolMessage 必须成对追加到历史，缺一不可

LangGraph 中通过 `add_messages` 注解自动维护这条消息链，确保每次 LLM 调用都能看到完整的推理历史。

## 16. Agent 系统中 LLM 对代码仅做了 RAG 查找吗？是否可以后训练增强代码能力

### 当前系统的做法

在我们的 5G 测试 Agent 系统中，LLM 对测试用例"代码"（YAML/Python 格式的用例脚本）的处理主要有两条路径：

1. **RAG 查找**（主路径）：通过 `Test_Case_Query` 工具检索相关用例，将用例内容注入上下文，LLM 读取后决定参数配置或触发执行。LLM 不直接生成执行脚本，而是生成工具调用参数。
2. **参数生成**（辅助路径）：LLM 根据用户描述和检索到的协议内容，生成 `Simulation_Runner` 的 `env_config` 参数（如 `{"band": "n78", "channel_model": "CDL-C"}`），本质上是结构化 JSON 生成，而不是代码生成。

### 是否可以做后训练

可以，但需要明确收益和成本：

**值得做后训练的场景**：
- 协议参数精确匹配率低（如模型生成的 `preambleTransMax` 数值范围不符合 3GPP 约束）
- 用例 YAML 格式错误率高（字段名拼写错误、必填项遗漏）

**推荐方案**：
```
数据来源：HITL 阶段专家修正的工具调用参数（专家改了 LLM 生成的 env_config）
训练方式：LoRA 微调（rank=16~32），不做全参数更新，节省显存
基座选择：在当前使用的推理模型基础上微调，避免引入新的部署复杂度
评测指标：协议参数精确匹配率（Exact Match）+ 用例 Schema 校验通过率
```

**不建议做的情况**：当前 RAG + Prompt Engineering 已能覆盖 90%+ 的用例，追求最后 10% 的准确率可以先考虑改进工具的 description 和示例，比做微调的 ROI 更高。

## 17. Multi Token Prediction 是怎么实现的

Multi Token Prediction（MTP）是 Meta 和 DeepSeek 等在研究中使用的技术，让模型在单次前向传播中同时预测未来多个 token，而不仅仅是下一个 token。

### 标准 Next Token Prediction 的局限

标准 LM 在位置 $t$ 只预测 $t+1$ 的 token：

```
输入：[x1, x2, x3, x4]
目标：[x2, x3, x4, x5]     ← 只预测 next 1 token
```

### MTP 的实现方式

在主预测头之外，额外添加 $D$ 个独立的预测头，每个头预测未来第 $k$ 个 token：

```python
# 伪代码结构
hidden_states = transformer_backbone(input_ids)   # 主干网络共享

# 主预测头（next-1 token）
logits_1 = lm_head_1(hidden_states)   # 预测 t+1

# 额外预测头
logits_2 = lm_head_2(hidden_states)   # 预测 t+2
logits_3 = lm_head_3(hidden_states)   # 预测 t+3

# 训练 loss = λ1*CE(logits_1, target_1) + λ2*CE(logits_2, target_2) + ...
```

关键设计：主干网络只跑一次（共享计算），额外的预测头参数量很小，训练额外成本可控。

### MTP 的两个作用

**训练阶段**：提供更丰富的训练信号，让模型每个位置同时学习短程和长程依赖，相当于数据增强，实验显示能提升模型在代码和推理任务上的准确率。

**推理阶段（结合 Speculative Decoding）**：
```
1. 用 MTP 头同时"草稿"预测 t+1, t+2, t+3
2. 主模型用一次前向验证这 3 个 token 是否可接受
3. 若全部接受，直接输出 3 个 token（相当于 1 次前向生成了 3 个）
4. 若第 2 个 token 被拒绝，只接受第 1 个，重新从第 2 个位置开始
```

理论上接受率越高，吞吐量提升越大（DeepSeek-V3 报告约 1.8~2x 吞吐提升）。

## 18. 微调 3B 模型的空间占用评估，KV Cache 占其中多大部分

### 模型参数占用

以 Qwen3-4B（实际约 4B 参数）为例：

| 精度 | 参数占用 |
|------|---------|
| BF16 / FP16 | 4B × 2 bytes = **8 GB** |
| INT8 量化 | 4B × 1 byte = **4 GB** |
| INT4 量化 | 4B × 0.5 byte = **2 GB** |

### 全参数微调的总显存

```
参数（BF16）：    8 GB
梯度：            8 GB   （与参数同精度）
Adam 优化器：    32 GB   （FP32 主权重 + 两个动量项各 16 GB）
激活值（batch=4, seq=2048）：约 4~8 GB

全参数微调合计：约 52~56 GB → 需要多卡
```

### LoRA 微调的总显存

```
冻结参数（BF16，不存梯度）：8 GB
LoRA 适配层参数（rank=16，约 0.1% 参数量）：~0.01 GB
LoRA 梯度 + Adam：~0.03 GB
激活值：4~8 GB

LoRA 微调合计：约 12~16 GB → 单卡 A30/A100 可以跑
```

### KV Cache 占多大

KV Cache 的计算公式（推理时）：

```
KV Cache = 2 × num_layers × num_kv_heads × head_dim × seq_len × batch_size × bytes_per_element
```

以 Qwen3-4B 为例（约 36 层，GQA num_kv_heads=8，head_dim=128）：

| 场景 | KV Cache 大小 |
|------|--------------|
| batch=1, seq=2048, FP16 | 2 × 36 × 8 × 128 × 2048 × 1 × 2 ≈ **0.29 GB** |
| batch=8, seq=2048, FP16 | 约 **2.3 GB** |
| batch=8, seq=8192, FP16 | 约 **9.2 GB** |

**结论**：微调时 KV Cache 占比很小（训练序列通常 2048 以内），主要开销在参数 + 梯度 + 优化器。推理时 KV Cache 随序列长度和 batch size 线性增长，长上下文场景下（seq=32K）可能超过模型参数本身的显存占用。

## 19. 如何处理 CoT 时正确率上升但时间变长的问题

这是精度-延迟的经典 tradeoff，有以下几种应对策略：

### 策略 1：Adaptive CoT（按难度决定是否开启）

不是所有问题都需要 CoT。对问题做难度分类：
- 简单问题（意图明确、参数直接）：直接输出答案，不开 CoT
- 复杂问题（多步推理、歧义消解）：才触发 CoT

实现方式：在 Agent 的路由节点加一个轻量分类器（或用 LLM 的 `thinking_budget` 参数），根据问题复杂度动态分配推理 token 上限。

### 策略 2：Budget Forcing（给推理设置 token 预算）

强制 CoT 在固定 token 数内结束。超出预算时插入强制结束符，让模型在有限步骤内给出最优猜测。DeepSeek-R1 中通过 `thinking` 和 `max_tokens` 控制推理上限。

### 策略 3：思维链蒸馏

用带完整 CoT 的大模型输出，训练小模型让其"内化"推理过程。训练好的小模型推理时可以直接给出答案，无需显式输出长思维链。蒸馏后的小模型保留了大部分推理准确率，但速度与直接推理相当。

### 策略 4：并行推理取最快

对同一问题并发发起多条推理路径，取最先完成的高置信答案。适合对延迟极敏感的场景，但会增加计算成本。

### 策略 5：Speculative Decoding 加速 CoT token 生成

结合 MTP 或草稿模型，加快 CoT token 的生成速度，在不减少推理步骤的前提下降低端到端延迟。

**在我们系统中的做法**：Adaptive CoT——Guardrail 节点（安全判断，快速）不用 CoT；结果判定节点（根因分析，需要推理）才用 CoT，并对 thinking token 设置上限（1024 token）。

## 20. 为什么不能用 ES 数据库去检索向量

### ES 的设计定位

ES（Elasticsearch）基于 **Apache Lucene** 的倒排索引，专为关键词的精确/模糊匹配设计：

```
倒排索引：词 → 文档列表（精确、快速）
BM25：词频 + 逆文档频率 → 相关性打分
```

### 向量检索的需求

向量检索需要在高维空间（128~1536 维）中找 **近似最近邻（ANN）**，核心数据结构是：
- HNSW（分层可导航小世界图）
- IVF + PQ（倒排文件索引 + 乘积量化）

这两种结构与倒排索引在设计上完全不同。

### ES 不适合向量检索的具体原因

1. **性能差距**：ES 虽在 7.x 后加入 `dense_vector` + kNN 支持，但底层 ANN 实现性能比 Milvus、Faiss 差。在百万级向量、高 QPS 场景下，Milvus 的延迟通常是 ES 的 1/3~1/5。

2. **不支持 GPU 加速**：Milvus 可以将 HNSW 索引加载到 GPU，大幅加速高维向量的并行距离计算；ES 只能用 CPU。

3. **资源调度模型不匹配**：ES 按分片（shard）管理数据，向量索引需要频繁的 segment merge，在向量更新频繁的场景下 IO 压力大、延迟抖动明显。

4. **标量过滤与向量检索的结合**：Milvus 支持先标量过滤（如只搜 `priority=P1` 的文档），再做向量 ANN 检索，两者原生集成。ES 的 pre-filter + kNN 实现在复杂过滤条件下准确率会下降。

### 实际系统的分工

```
ES（BM25）：负责关键词精确匹配，处理协议编号、参数名、专有词
Milvus：   负责语义向量检索，处理语义相似度
RSF 融合：  两路结果合并打分，取长补短
```

两者不是替代关系，是互补关系。

## 21. User Prompt 和 System Prompt 的区别，Agent 系统中哪些放在 User Prompt

### 本质区别

| 维度 | System Prompt | User Prompt |
|------|--------------|-------------|
| 角色 | 模型的"行为规范"，由开发者设定 | 当前用户的具体输入 |
| 变化频率 | 整个对话中固定不变（或每轮固定） | 每轮对话动态变化 |
| 可信度 | 更高（模型被训练为优先遵守） | 较低（视为外部输入） |
| Prompt Cache | 适合缓存（不变内容命中率高） | 不适合缓存 |

**安全注意**：Prompt Injection 攻击就是试图通过 User Prompt 中注入恶意指令来覆盖 System Prompt 的约束。System Prompt 的权威性更高，但不是绝对安全。

### Agent 系统中的分配原则

**放在 System Prompt 中**：
- 角色定义（"你是一个5G无线测试助手"）
- 工具使用规范（"调用工具前先明确当前 Thought；不得直接猜测测试结果"）
- 输出格式约束（Thought/Action/Observation 格式，或 JSON Schema 约束）
- 安全约束（"不得生成批量删除、修改基站配置的工具调用"）
- 全局背景（项目架构、团队分工的静态背景）

**放在 User Prompt 中**：
- 当前用户的自然语言请求
- RAG 检索到的上下文（动态的，每次查询不同）
- ToolMessage 的执行结果（Observation，追加到消息历史）
- 当前轮次的对话历史摘要（Context 过长时压缩后追加）

**为什么 RAG 检索结果放 User Prompt 而不是 System Prompt**：

RAG 结果是动态的（每次查询不同），放 System Prompt 意味着每次都需要重建完整的 System Prompt，无法命中 Prompt Cache（缓存按前缀匹配，System Prompt 一变则全部失效）。将动态内容放 User Prompt，System Prompt 的前缀保持不变，Cache 命中率高，可以显著降低 Token 计算成本。

## 22. 千问3 的归一化是怎么做的

Qwen3 在归一化上的设计与当前主流大模型（Llama、Mistral）保持一致，核心有以下几点：

### 使用 RMSNorm 而非 LayerNorm

**LayerNorm** 计算每个 token 的均值和方差，再做标准化：

$$\text{LayerNorm}(x) = \frac{x - \mu}{\sigma} \cdot \gamma + \beta$$

**RMSNorm** 只计算 RMS（均方根），去掉均值计算，去掉 bias 项：

$$\text{RMSNorm}(x) = \frac{x}{\sqrt{\frac{1}{d}\sum x_i^2 + \epsilon}} \cdot \gamma$$

优点：计算量更小（省去均值统计），效果与 LayerNorm 相当，是现代大模型的标配。

### Pre-Norm 架构

Qwen3 采用 **Pre-RMSNorm**（在 Attention 和 FFN 之前归一化），而不是 Post-Norm：

```
Pre-Norm（Qwen3、Llama 等）：
  x_out = x + Attention(RMSNorm(x))   ← 先 Norm，再加残差

Post-Norm（早期 BERT/GPT）：
  x_out = RMSNorm(x + Attention(x))   ← 先加残差，再 Norm
```

Pre-Norm 的梯度流更稳定（残差路径上没有 Norm 阻断梯度），训练更容易，是目前的主流选择。Post-Norm 理论上最终效果更好，但训练需要精细的学习率预热。

### Embedding 层后的额外 RMSNorm

Qwen3 在 token embedding 之后、进入第一个 Transformer Block 之前，额外加了一层 RMSNorm，用于稳定初始输入的尺度。不是所有模型都有这个设计，Qwen 系列是有的。

### 无 Bias（Weight-Only）

所有 Linear 层和 RMSNorm 均去掉 bias 项，减少参数量，同时在量化时更友好（有 bias 的 Linear 层在 INT8 量化时需要额外处理 bias 的精度）。
