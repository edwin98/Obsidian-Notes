# 5G无线网络智能测试验证 Agent 系统 - 技术深度报告 (面试专用)

## 一、 项目背景与业务挑战

### 1. 业务痛点
随着5G基站的大规模商用，网络验收与回归测试面临着严峻的挑战：
- **用例规模庞大**：测试用例数量超过十万条，覆盖信道、切换、干扰、容量等多维复杂场景。
- **效率瓶颈**：传统的纯人工执行方式效率极低，单轮回归测试动辄需要数天时间，极大阻碍了持续集成和交付（CI/CD）的流转。
- **标准化缺失**：测试结论的判定高度依赖于领域专家的经验（如日志分析、信令交互细节、KPI趋势判断），难以形成标准化的闭环，从而导致漏检和误判。

### 2. 核心目标
研发一套基于大语言模型（LLM）驱动的自主网络测试 Agent，实现**从测试需求解析 $\rightarrow$ 用例生成 $\rightarrow$ 自动执行 $\rightarrow$ 结果智能判定**的端到端全自动闭环，大幅降本增效，提升测试覆盖率与准确率。

---

## 二、 核心技术栈解析 (Tech Stack)

本项目构建于业内领先的 LLM 工程化技术栈之上，打通了从模型推理到业务编排的全链路：

- **大语言模型基座**：**Qwen3-32B** (阿里云开源大规模语言模型)，在指令遵循、长文本理解和垂直领域推理上具有极高的性价比，作为系统的主要“大脑”。
- **Agent 编排框架**：**LangChain / LangGraph**。利用 LangChain 标准化工具接口，基于 LangGraph 构建基于图谱的状态机（State Graph），实现具有循环、条件分支和记忆能力的复杂多步 ReAct 逻辑。
- **高性能推理服务**：**vLLM**。基于 PagedAttention 显存优化技术，解决高并发下的 KV Cache 显存碎片问题，大幅提升 Qwen3-32B 在企业级并发下的吞吐量。
- **RAG 向量与检索引擎**：**Milvus** (HNSW 向量检索) + **Elasticsearch** (BM25 倒排索引)。构建混合检索架构，用于历史缺陷库、通信协议文档的精准召回。
- **异步任务队列**：**Celery + Redis / Kafka**。应对长时间的仿真环境运行和测试执行任务，将 Agent 推理过程与后台测试环境解耦。
- **在线服务框架**：**FastAPI**。构建高性能的基于 ASGI 的异步接口，并利用 SSE (Server-Sent Events) 协议实现打字机流式输出和任务状态实时推送。

## 三、 系统架构与关键技术细节落地

### 1. ReAct 多工具 Agent 核心框架实现
为应对复杂的网络测试依赖图，系统采用了 **ReAct (Reasoning and Acting)** 范式架构，并由 **LangGraph** 提供底层状态流转支持。
- **工具编排 (Tool Orchestration)**：
  定义并注册了四类核心工具供 LLM 调用：
  1. `Test_Case_Query`：结构化用例查询工具，对接底层用例数据库，支持按场景、频段、特性检索。
  2. `Simulation_Runner`：仿真环境调用工具，通过 API 触发底层5G仿真仪开始下发信令。
  3. `Metrics_Collector`：指标采集工具，通过 SSH/RPC 从网元实时拉取性能日志、抓包数据。
  4. `Baseline_Comparator`：基线对比工具，调取历史安全版本的 KPI 波动曲线进行统计学比对。
- **LangGraph 状态机流转设计**：
  在 LangGraph 中定义了一个 `AgentState` 结构，包含 `messages` (对话历史), `current_step` (当前执行步骤), `tool_outputs` (工具返回缓存) 以及 `error_count` (连续错误计数)。每次 LLM 思考后，决定进入 `Action` 节点还是 `Finish` 节点。
- **双重熔断机制 (防幻觉与死循环机制)**：
  - **最大图迭代深度限制**：在 LangGraph 路由节点拦截，当单任务流转的 recursion_limit 超过设定的阈值（如15跳）时强行挂起并上报异常。
  - **工具调用置信阈值拦截**：通过约束 LLM 在输出 Action 规划时同步输出置信分 `Confidence: 0.0~1.0`。若连续两次决策置信度低于 0.6，则触发降级流程，调用人工复核接口 (Human-in-the-loop)。

### 2. RAG 赋能下的 LLM 用例生成
自动生成符合严格 5G 协议规范的测试用例是本系统的难点。使用 Qwen3-32B 结合 RAG 实施补全：
- **混合检索策略**：基于通信规范（3GPP 文档汇编）以及企业历史 Bugzilla 缺陷库，建立专业语料库。利用 ES(BM25) 进行包含专有通信代号（如 RSRP, SINR, CQI）的精准关键字匹配，利用 Milvus 进行深度语义向量匹配。通过自研融合算法结合两路召回，并用 Cross-Encoder 进行语义重排（Rerank）。
- **历史缺陷边界补全**：单纯依赖基座难以生成“极端干扰条件”等边缘用例。系统自动检索出同类功能的“历史遗留漏测 Bug”，将当时的触发条件打包成 RAG Context 喂给 Qwen3。使得生成的用例天生具备“防覆辙”能力。
- **用例回流飞轮 (Data Flywheel)**：结构化的测试用例生成后，会进入待审核队列。测试专家确认或微调通过的用例被自动转化为新的 Embedding 索引块存入 Milvus。系统越用越聪明。

### 3. 全自动执行与结果智能判定 (双轨机制)
用例执行完毕后，海量日志与 KPI 如何转化为明确的 PASS / FAIL 结论？
- **第一轨：硬规则与基线对比 (统计轨)**
  利用传统统计算法计算测试期间内核心 KPI (如下行吞吐量掉线率) 的均值、方差。与同等条件下的历史 Baseline 采用假设检验 (如 T-Test、KS-Test) 或简单的阈值包络策略进行刚性对比。违背容忍度底线者直接判定 FAIL。
- **第二轨：LLM 日志语义深度解析分析轨**
  对于 KPI 波动不明显，但存在隐性协议异常的场景。提取系统底层日志或 PCAP 抓包转译通过大模型进行理解。Qwen3 展现了卓越的代码信令日志分析能力，能够分析出类似“UE RRC Setup Request 在定时器 T300 超时未收到基站回复”的微观错误。
- **聚合与根因分析 (RCA - Root Cause Analysis)**
### 4. Human-in-the-loop (HITL) 机制：高风险用例识别与人工接管
纯粹让 Agent “放飞自我”在严肃的企业级通信测试场景中是不切实际且极其危险的。特别是在自动生成测试用例并下发执行的环节，一旦生成包含破坏性动作的异常用例（如整网核心参数重置、超大并发 DDoS 式信令压测），极易导致现网或仿真环境的灾难性大面积宕机。因此，我们在核心链路中深度构建了**基于“高风险高敏用例识别”的 HITL 人工接管安全防线**。

#### (1) 技术栈与工程落地架构
- **独立高危意图识别护栏 (Guardrail Classifier)**：
  我们并不完全依赖进行推理和生成的主干模型（Qwen-32B）做自我裁判，以防止“既当运动员又当裁判”的逻辑盲区。在下发仿真器之前的核心关卡上，串联了一套轻量级的安全对齐护栏（参考 `Nemo-Guardrails` 与 `Llama-Guard` 思想机制）。系统会对大模型输出准备下发的 `Command Payload` 进行“正则规则库词频（如 `format`, `reset_all`, `force_reboot` 等敏感词拦截）+ 轻量级分类模型”双缝检测。一旦识别到高危的高敏意图动作，立即触发硬熔断。
- **状态冻结与中断框架 (LangGraph Checkpoint & Interrupts)**：
  传统 While 循环架构如果在执行中断点等待人员审批，会导致进程长期阻塞、内存泄漏或 HTTP 504 超时异常。我们在 LangGraph 编排中运用了原生的持久化组件 `Postgres Saver (Checkpoint)` 。我们在风险评估节点或执行节点（如 `Node_Execute_Simulation`）外侧包裹了 `interrupt_before` 钩子。触发高危预警后，由 LangGraph 将当前所有的上下文、推导过程（`State`）快照序列化到数据库并终止该子进程运算，彻底释放显存和计算资源。
- **全异步链路的唤醒与修正 (Celery + Webhook + SSE 流推)**：
  任务挂起的同时，Celery 异步队列会触发告警 Webhook 发往通信群（如钉钉）。业务专家在前端界面看到高危被拦截的用例清单，在前端可以直接 Review 这段生成的测试代码，手动修改异常参数（如把无限并发数 $N\rightarrow \infty$ 改为限流的 10）。确认“Approve”后，前端调用唤醒 API，后端去 Postgres 库拉取 `Thread_ID` 对应的序列化状态并在原语境下重构，将人员干预修复后的最新 Payload 无缝注入其中，随后调用 `.resume()` 唤醒原 Agent 任务，继续下游环节流转。

#### (2) 护栏与接管背后的深度业务思考
1. **防止灾难性环境摧毁与业务底线兜底**：  
   **业务思考**：5G 协议极其脆弱。在大模型特有的幻觉（Hallucinations）发作时，往往能一本正经地写出法理通过但业务致命的参数配置修改脚本（比如将周边邻区信号阈值拉穿导致所有基站掉线）。
   **业务落地**：通过高危动作阻断（包含底层写权限、删除权限的信令），系统为脆弱的底层硬件设立了最后一道物理级别的安全隔离墙。如果在 HITL 阶段被专家打回（Reject），专家的“批改原因”会作为一轮 `Observation` 反供给 LLM 迫使其反思“自己为什么被拦截”，促使其在下一次重新生成用例规避此类错漏。
2. **“毫无历史包袱的新用例”确权与知识飞轮**：
   **业务思考**：我们有时会发现 Agent 借助 RAG 上下文“灵光一闪”，拼凑生成出了向量库（Milvus）里所有历史用例相似度都极低的新奇剧本（Novel Case）。对于这种完全脱离经验背书的测试用例，系统“不敢”直接跑，但不跑就意味着浪费了发现新缺陷的极大可能。
   **业务落地**：触发“低相似度” HITL 隔离。测试专家介入，若发现该逻辑闭环精湛、具备极高隐蔽 Bug 挖掘潜力，便直接拍板执行，并附上“Golden”标签。这些经过专家确权的金标新用例会立马回流入知识库做 Embedding，使得后续类似场景直接拥有了历史背书。这正是本系统**越用越聪明、测试边界自我拓张**的核心数据飞轮（Data Flywheel）发动机。
3. **“自我怀疑”阶段的信任降级（防止误判）**：
   **业务思考**：测试系统最怕的不是报错，而是机器自作主张把一个其实是 FAIL（偶发掉网） 的结果误判为 PASS（吞没故障）。基于大模型做结果研判时，有时会碰到复杂的 KPI 结合信令波动的模棱两可现象。
   **业务落地**：我们在底座强制要求 LLM 吐出置信度分段 `Confidence Score`。当掉入设定的灰度区间（例如 0.60~0.65 之间，即“可疑但无绝对硬指标违背证据”时），模型自爆不确定性，强制拦截进 HITL 给人员会诊确认。更关键的是，专家接审最终的研判结果（真实 Label），会立刻转化为一份高质量 DPO / SFT 偏好对齐语料积攒下来。系统通过一次次的专家兜底救场，反向从专家手里收集了世界上最优质的模型微调训练集。

### 5. 模型后训练与强化学习闭环 (SFT & RLHF/DPO)

通用大模型（如 Qwen3-32B）虽然具备强大的逻辑推理能力，但在高度垂直的 5G 通信测试领域中，仍面临**领域黑话理解不足、结构化格式（JSON/YAML）遵循不稳定、易产生危险逻辑幻觉**三个核心问题。因此，对 Agent 的基座模型进行后训练（Post-Training）、微调（Fine-Tuning）与偏好对齐（Alignment）是系统从可用走向高可靠的必然选择。

本系统构建了基于 **HITL 数据回流的持续微调飞轮**，具体分为两个阶段：

#### (1) SFT (Supervised Fine-Tuning) 监督微调：解决领域知识与格式遵从
- **为什么需要 SFT**：零样本下，基座模型很容易将特定的通信频段张冠李戴，或是生成的仿真代码在结构上缺少必填字段（如 `bearer_id`, `rnti` 等）导致下游执行器解析崩溃。仅靠 Prompt 与 RAG 上下文，不仅稳定性差且极易耗尽上下文 Token 限制。
- **如何进行**：
  - **语料来源**：从早期系统的 HITL 环节中沉淀高质量的 **Golden Dataset (金标数据集)**（约万级规模），主要来源于测试专家亲自编写、或由专家人工审核修改通过后的测试用例与故障排查报告。
  - **训练目标**：构建 `(Instruction, Response)` 数据对。对基座进行 SFT 训练，将强约束的行业语料和通信参数格式深度内化到模型权重中，建立模型对领域基础概念的肌肉记忆。

#### (2) DPO (Direct Preference Optimization) 偏好对齐：有效抑制幻觉与界定安全边界
- **为什么需要 DPO/RLHF**：在网络测试这种严肃场景下，“凭空捏造一个带有‘重置全部基站’等高危参数的用例”比“不生成用例”更可怕。普通的 SFT 只能教模型“什么是对的规范”，但很难教它“什么是危险/被拒绝的边界”。我们需要通过偏好学习来对齐人类网络安全的底层直觉。
- **如何进行**：
  - **偏好对数据的天然收集**：HITL 组件是我们天然的 DPO 数据采集器。Agent 初次生成的高危漏洞用例被测试专家打回拦截（Reject），这就构成了 $Y_{rejected}$ 样本；专家手工修改好安全参数后重新提交通过的用例，则构成 $Y_{chosen}$ 样本。
  - **DPO 训练过程**：摒弃传统 RLHF 复杂度高的 Reward Model 训练，直接采用轻量的 DPO 算法。利用 `(Prompt, Y_chosen, Y_rejected)` 三元组对 SFT 后的模型进行直接对齐训练，加大合法信令组合的生成概率梯度，严厉惩罚和打压高风险破坏性动作词汇的权重。
- **业务收益**：经过 DPO 对齐后，模型对于“高危不确定场景”会自动触发自我怀疑机制（表现为 Confidence Score 跌出阈值），灾难性幻觉生成率和触发系统高危拦截护栏的频率显著降低，系统真正具备了“企业级安全对齐”能力。

## 四、 工程化部署与高并发性能优化

大型 Agent 系统的落地，不仅是算法层面的工作，更是工程系统挑战：
- **大模型并行推理加速 (vLLM Deploy)**：基于 vLLM 部署 Qwen3-32B，开启 PagedAttention 技术进行动态显存分页调度，使得测试批发出高并发请求时，系统的 KV Cache 重用率大幅提升，保证了在线评判模块 30+ Tokens/s 的稳定生成速率。
- **长时任务的异步化解耦 (FastAPI + Celery)**：测试执行通常耗时几分钟到数小时不等。前端使用 FastAPI 接收测试请求后，将复杂的“长图执行逻辑”打包下发至 Celery 异步工作队列。用户不被阻塞，系统可通过 Server-Sent Events (SSE) 长连接将节点跳跃状态（如“正在执行仿真...” “正在分析日志...”）实时打向前端界面。

## 五、 业务价值与量化成果

- **覆盖度飞跃**：在涵盖 **1200个典型 5G 场景**的验证与回归测试集上，系统将原本无法自动化的灰度用例激活，整体自动化率达 **91%**。
- **分析精准**：利用双轨机制，机器判定的 PASS/FAIL 准确率达 **89%**，人工仅需复检极少数难点用例以及确报缺陷。
- **效率突破**：单轮基线回归测试周期从 **纯人工执行的 3天 (72小时)** 极速压缩至 **4小时以内**。
- **质量兜底**：过去由于人工疲劳产生的漏测问题由于防呆与历史 RAG 库补全，导致线上版本缺陷漏检率（缺陷逃逸率）断崖式下降 **60%**。本系统成为部门自动化流水线的基础设施引擎。

## 六、 面试高频考察点预判与高分答题策略

为了在面试中全方位展现“大模型算法工程师”的技术实力，建议参考以下话术框架应对深度提问：

### Q1: 为什么要选择用 LangGraph 而不是基础的 LangChain Agent 架构？
**高分答题策略**：
“因为我们的测试业务场景是一张复杂的图，而不是单一的线。传统的 LangChain Agent (比如 `AgentExecutor`) 本质是一个死循环黑盒，对于一个一旦出错就可能连锁崩溃的通信自动化测试任务来说，太不可控了。”
“采用 LangGraph，我们将整个测试和诊断过程抽象为一个**状态机（State Machine）**。这有三大好处：**第一是容错性**，我可以明确定义异常重试路径，并在图中挂载人工介入（Human-in-the-loop）断点；**第二是多分支并发**，比如在诊断结果阶段，统计基线比对与 LLM 语义分析可以作为并行节点被激活；**第三是长期记忆持久化**，LangGraph 原生支持 Checkpoint 机制，测试被意外中断后可以随时从断点图状态恢复，这在数小时的长线测试任务中是致命需求。”

### Q2: Qwen3-32B 在生成结构化用例和底层日志分析上表现如何？你们怎么做提示词工程和微调的？
**高分答题策略**：
“大模型由于训练语料的通用性，非常容易在通信领域（3GPP标准）产生幻觉。最初用指令直接生成时，往往出现格式不符合我们测试平台规范或参数张冠李戴（比如把 5G 的专有频段代入 4G 参数）的问题。”
“我们首先依靠 **RAG 强约束**，在 Prompt 注入模板化的历史规范字典。而在进阶阶段，我们提取了人工审核通过好的 **Golden Dataset**（约 5 万对齐数据），对 Qwen3 进行了 **SFT（监督微调）**。重点加强了其 `JSON/YAML` 等结构化代码输出格式的遵从遵循度，同时专门引入了各种网络乱码错误日志数据进行特征学习。使得它看到乱七八糟的基站报错底层堆栈时，能稳定按 `[根本原因] + [信令级异常点]` 的结构输出诊断描述。”

### Q3: 混合检索中，你如何权衡 Elasticsearch(BM25) 与 Milvus(稠密向量) 的比重？
**高分答题策略**：
不要说固定的比重（如简单的50/50）。这是一个工程和算法结合的难点。
“我们在实际投产中发现，通信领域的专业词汇（例如 `PRACH`, `BWP`, `N2 interface`）存在严重的 **OOV (Out-Of-Vocabulary)** 问题，纯靠 Embedding 模型很容易丢失这些高辨识度的‘骨架词’，产生语义漂移。因此采用**多路并发召回 + 自适应重排**。基于 BM25 的 ES 负责保底，确保专有代号和缩写的命中率；基于 BGE/MTEB 霸榜的 Embedding 模型存入 Milvus，负责捕捉长难句的用户提问意图。最终多路合并后，统一送给诸如 `bge-reranker` 等专门重排模型，进行 Cross-Attention 前端交叉交互打分，截取出最精准的 Top-K 形成 RAG 上下文。”

### Q4: 你们的双重熔断机制是怎么具体实施的呢？置信度阈值是怎么来的？
**高分答题策略**：
“双重熔断本质是在**系统级层面防御大模型的幻觉递归**。第一层是 `max_iterations` 的硬拦截，LangGraph 中的图深度限制，比如 15 步没有走到 Finish 就报错截断。第二层是软性拦截机制。我们会设计一个专用的 `Evaluation Node` 或者在底层提示词中规定模型必须输出一个 `<confidence_score>`（并且用少样本提示告诉它什么样的答案是高分，什么样由于证据不足是低分）。我们在工程上经过压测和调优，发现了比如 `0.65` 是一个敏感的相变点。低于此值，Agent 自主执行极易把系统搞挂，系统将停止调用并把当前的中间状态与思考逻辑发送到钉钉群，由测试专家确认，并以此触发 HITL 人机协同处理。”

### Q5: 你提到系统中加入了 Human-in-the-loop (HITL)，请问在工程架构上具体是怎么落地的？能够解决业务上的什么问题？
**高分答题策略**：
“Agent 在工业落地时最大的阻碍常常是业务方的‘信任问题’。我们的思路是**从 Copilot (辅助副驾) 逐步自然过渡到 Autopilot (自动驾驶)**，这就必然依赖 HITL 机制作为桥梁。”
“在纯**工程架构落地**层面，这是非常依赖于 **LangGraph 的 Checkpoint 状态持久化机制**。过去如果是写死 While 循环的 AgentExecutor，等待专家审批这几个小时，会导致线程持续阻塞或 Http 超时。但我们是在流程图的高危操作节点前配置了 `interrupt_before` 断点。当触发审核时，把此刻所有的中间参数、对话上下文原样序列化写入持久化存储层（比如 Postgres），而后直接释放计算资源。业务方在网页端点击同意后，系统通过 API 唤醒，从 Checkpoint 重建整个状态机重走剩余流程。”
“这为我们实现了三大**业务兜底能力**：首先是用例生成环节，防范有毒数据污染我们的知识大盘；其次是在结果判定模块中处理‘机器自己都怀疑自己’的灰度边缘故障；最后则是为‘高权限动作环境操作’建立了最后一道物理安全隔离墙，保障测试现网不出事故。可以说 HITL 是 Agent 摆脱纯玩具属性转向工业生产力的基石。”

### Q6: 你们的 Agent 系统在实践中需要做后训练（Post-Training）或微调吗？如果是，具体是怎么做的？
**高分答题策略**：
“必须做。通用基座大模型就像一个学识渊博但缺乏领域经验的实习生，在严苛的 5G 通信测试环境中，如果只靠 RAG 和系统 Prompt 约束，它不仅输出的测试脚本格式容易错乱缺失字段，更可怕的是极易产生越权尝试和危险信令操作的‘逻辑幻觉’，这在工业落地是不能容忍的。

针对这个问题，我们在框架中设计了**基于‘人机协同（HITL）’的数据反哺后训练飞轮**，分为两步：
**第一步是 SFT（监督微调）解决格式和术语盲区**。我们将早期通过人员审核校对的数万条极高质量的规范测试代码和专家级诊断报告作为 Golden Data 进行全参/LoRA微调。将极其复杂的信令层级格式、5G 专有行业词汇压缩成模型的‘肌肉记忆’，它现在能闭着眼按格式吐出正确的配置文件。
**第二步也是最关键的，利用 DPO（直接偏好优化）进行高危风险对齐抑制幻觉**。我们充分榨取了 HITL 人工拦截组件的价值：凡是被测试专家主动拦截并打回的安全不合规的 Agent 操作序列，直接作为负向参考样本（Rejected）；专家修改正确后的安全序列，作为正向参考样本（Chosen）。我们用这些人类安全偏好数据直接通过 DPO 算法对模型进行二次拉伸和对齐惩罚。
这套‘执行 -> 拦截打回 -> 专家批改 -> SFT/DPO 微调迭代’的闭环机制，使得我们的 Agent 的安全意识和判断越来越接近顶级的专家经验，灾难性错误生成率基本清零。”
