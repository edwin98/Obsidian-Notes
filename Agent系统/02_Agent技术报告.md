# 5G 无线网络智能测试验证 Agent 系统 — 技术深度报告

---

## 一、项目背景与业务挑战

### 业务痛点

5G 基站大规模商用后，网络验收与回归测试面临三个核心矛盾：

- **规模庞大**：测试用例超过 2 万条，覆盖信道、切换、干扰、容量等多维场景，人工全量执行单局点需要 3 天
- **标准化缺失**：测试结论判定高度依赖领域专家经验（日志分析、信令细节、KPI 趋势判断），难以形成标准闭环
- **漏测与误判**：人工疲劳、经验差异导致的漏测问题在版本迭代中持续累积

### 核心目标

研发一套 LLM 驱动的自主网络测试 Agent，实现从**测试需求解析 → 用例生成 → 自动执行 → 结果智能判定**的端到端全自动闭环，并通过 HITL 机制保障工业级安全可靠性。

## 二、系统设计哲学：三个嵌套闭环

理解本系统的关键，是把它看作**三个运行在不同时间尺度上的嵌套闭环**，而不是孤立的技术组件堆叠。

```
┌─────────────────────────────────────────────────────────┐
│  闭环 3：学习飞轮（离线，天/周级）                         │
│  HITL 数据 → SFT/DPO → 模型更安全；Golden Case → RAG 更准 │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  闭环 2：安全围栏（请求级，秒~小时）               │    │
│  │  置信度熔断 / 高危拦截 → HITL → 人工修正后恢复    │    │
│  │                                                  │    │
│  │  ┌─────────────────────────────────────────┐    │    │
│  │  │  闭环 1：ReAct 推理循环（毫秒级）          │    │    │
│  │  │  Agent → Guardrail → Tool → Observation  │    │    │
│  │  └─────────────────────────────────────────┘    │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

- **闭环 1** 解决"怎么完成一次任务"（毫秒到分钟）
- **闭环 2** 解决"怎么保证任务安全可控"（分钟到小时，含人工等待）
- **闭环 3** 解决"怎么让系统越用越聪明"（天到周，持续微调）

## 三、技术架构

### 3.1 技术栈

| 层次 | 组件 | 职责 |
|:---|:---|:---|
| **模型层** | Qwen3-32B（主干推理）/ 小模型（Guardrail 分类） | LLM 推理、安全检测 |
| **编排层** | LangGraph（状态机）/ LangChain（工具抽象）| Agent 图控制流 |
| **推理服务** | vLLM + PagedAttention | 高并发 KV Cache 优化 |
| **知识层** | Milvus（HNSW 向量）+ Elasticsearch（BM25）+ Cross-Encoder Reranker | 三级混合检索 |
| **异步层** | Celery + Kafka | 长时任务解耦、HITL 告警 |
| **接口层** | FastAPI + SSE | 实时状态流推送 |
| **持久化层** | Postgres（Checkpoint）/ Redis（缓存） | 状态快照、断点恢复 |

### 3.2 LangGraph 状态机图结构

系统将整个测试流程抽象为一张有向图，而不是简单的 `while` 循环。这是与教科书式 ReAct 最关键的工程区别。

```
START
  │
  ▼
[Agent 节点] — LLM 推理决策（Qwen3-32B）
  │
  ├─── 有工具调用 & 错误未超限 ──► [Guardrail 节点] — 安全关键词检测
  │                                      │
  │                             无高危 ──┤── 高危 ──► [HITL 节点]
  │                                      │                │
  │                                      ▼                │ 人工 Approve
  │                               [Tool 节点] — 执行工具   │
  │                                      │                │
  │◄─────────── Observation 追加至 State ─┘◄───────────────┘
  │
  ├─── 无工具调用 & 置信度正常 ──► [结果判定节点] — 双轨判定
  │                                      │
  │                              置信度≥0.65 → END
  │                              置信度<0.65 → [HITL 节点]
  │
  └─── 错误超限 / 置信度过低 ──► [HITL 节点]
```

**核心 State 字段**：

```python
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # 对话历史（自动追加）
    current_step: str                        # 当前执行阶段
    tool_outputs: dict                       # 工具返回缓存，供结果判定使用
    error_count: int                         # 连续工具错误计数（软熔断）
    confidence_score: float                  # 当前决策置信度
    hitl_required: bool                      # 是否需要人工介入
    hitl_feedback: str                       # 人工审核后的反馈文本
    final_result: Optional[str]              # 最终判定结果（JSON 字符串）
```

### 3.3 工具链

| 工具 | 职责 |
|:---|:---|
| `Test_Case_Query` | 按场景/频段/特性从知识库检索测试用例 |
| `Simulation_Runner` | 触发 5G 仿真平台执行测试 |
| `Metrics_Collector` | 通过 SSH/RPC 从网元实时拉取 KPI |
| `Baseline_Comparator` | 与历史安全版本做统计对比（T-Test / KS-Test）|
| `Log_Analyzer` | 解析 RRC/NAS/PDCP 信令日志，识别协议级异常 |
| `Fleet_Manager` | 多局点并发探针集群调度（全网验收场景）|

## 四、关键技术模块详解

### 4.1 RAG 赋能的 LLM 用例生成

通信领域的 OOV（Out-of-Vocabulary）问题严重：`PRACH`、`BWP`、`N2 interface` 等专有名词纯靠 Embedding 模型容易产生语义漂移。系统采用三级混合检索：

```
用户查询 / 用例生成需求
      │
      ├──► ES (BM25) — 精确命中通信专有名词、缩写
      │
      └──► Milvus (HNSW) — 语义向量捕捉长难句意图
                │
                ▼
         自研 RSF 融合算法（按问题长度自适应调权）
                │
                ▼
         Cross-Encoder Reranker + 动态阈值过滤
                │
                ▼
           Top-K Context → 注入 LLM Prompt
```

**知识库构成**：
- 3GPP 协议文档（BM25 精确匹配协议规范）
- 历史 Bugzilla 缺陷库（向量检索补全边界场景）
- 专家确认的 Golden Cases（带背书的高质量样本）

**检索前置增强**：
- Qwen3-4B 执行**指代消解**与**多路问题扩展**，提升召回覆盖率
- AutoPhrase 构建专有分词词典，解决通信术语错误切分问题

**数据飞轮**：专家确认后的用例自动 Embedding 入库，知识边界自我拓展。

### 4.2 Guardrail 安全节点

Guardrail 节点是本系统与教科书 ReAct 最核心的工程差异。工具调用在执行前**必须经过安全检测**，而不是直接下发。

**检测机制**（双缝检测）：
1. **规则层**：正则扫描工具参数，匹配高危关键词（`format_c`、`reset_all`、`force_reboot`、`wipe` 等）
2. **分类层**（可选）：轻量级安全分类模型二次确认（防止关键词绕过）

**为什么不让主干模型自己判断**：避免"既当运动员又当裁判"的逻辑盲区。生成幻觉的模型不能同时作为自己输出的安全裁判。

### 4.3 双重熔断机制

| 层级 | 触发条件 | 实现方式 | 处置 |
|:---|:---|:---|:---|
| **硬熔断** | 图迭代深度 > 阈值（如 15 跳）| LangGraph `recursion_limit` | 强制挂起，上报异常 |
| **软熔断** | 连续工具错误 ≥ 3 次，或 `confidence_score` < 0.65 | State 字段路由判断 | 路由至 HITL |

**置信度阈值 0.65 的来源**：通过压测找到相变点——低于此值时 Agent 自主执行错误率急剧上升，不是拍脑袋定的。

### 4.4 Human-in-the-Loop（HITL）工程实现

HITL 的工程难点：**如何在不阻塞服务器线程的前提下等待人工审批（可能长达数小时）？**

```
传统方式（错误）：
While 循环等待 → 线程持续阻塞 → 内存泄漏 / HTTP 504

LangGraph 方式（正确）：
触发高危 / 低置信度
  │
  ▼
interrupt_before 钩子：State 快照序列化写入 Postgres
  │
  ▼
释放计算资源（进程终止，GPU 显存释放）
  │
  ▼
Celery 异步触发钉钉 Webhook 告警
  │
  ▼
专家 Web 前端 Review → 修改参数 → 点击 Approve
  │
  ▼
API 唤醒 → 从 Postgres 拉取 Thread_ID 对应 State → .resume()
```

**HITL 解决的三类业务问题**：

1. **灾难性环境防护**：拦截包含高危写权限信令的用例，为脆弱的底层硬件设立最后防线
2. **模糊边界判定**：置信度处于灰度区间（0.60~0.65）时交由专家会诊
3. **Novel Case 确权**：与知识库相似度极低的新奇用例，由专家鉴别价值后决策是否执行——确认后的 Golden Case 自动回流知识库

**HITL 的战略价值**：它同时是数据飞轮的采集入口——被打回的异常用例成为 DPO 负样本，专家修正后的版本成为正样本，系统在安全拦截的同时持续收集微调数据。

### 4.5 结果判定双轨机制

单纯依赖 LLM 判定存在幻觉风险，单纯依赖统计规则无法处理复杂信令异常，因此采用双轨并行：

```
测试执行完成
  │
  ├──► 统计轨（硬规则）                  ├──► 语义轨（LLM）
  │    KPI 均值/方差                     │    PCAP 信令解析
  │    T-Test / KS-Test                  │    日志语义理解
  │    阈值包络对比                       │    根因关联推理
  │    → 明显违规直接 FAIL               │    → 隐性协议异常诊断
  │                                      │
  └──────────────── 综合判定 ────────────┘
                       │
              ┌────────▼────────┐
              │  置信度评分输出  │
              │ ≥0.65 → 自动结论 │
              │ <0.65 → HITL    │
              └─────────────────┘
```

**统计轨具体实现**：KPI 均值/方差与历史 Baseline 采用 T-Test / KS-Test 或阈值包络策略对比，违背容忍底线直接判 FAIL。

**语义轨具体实现**：Qwen3-32B 分析底层 RRC/NAS/PDCP 信令日志（PCAP 转译），能识别如"UE 上报 MeasurementReport，基站下发 RRCReconfiguration，但 SN Status Transfer 出现丢包超时"这类微观协议异常。

## 五、模型后训练闭环（SFT + DPO）

通用基座模型在 5G 测试领域面临三个核心问题：**领域术语理解不足**、**结构化格式输出不稳定**、**危险逻辑幻觉**。后训练是从"可用"走向"高可靠"的必经之路。

### 5.1 SFT 监督微调（解决格式与术语）

**问题**：零样本下，基座模型容易将通信频段张冠李戴，或生成的仿真代码缺少必填字段（`bearer_id`、`rnti` 等）导致执行器解析崩溃。

**数据构建**：
- Self-Instruct / Magpie 合成流水线：给强 LLM 注入 3GPP 协议核心段落，生成负面排障场景
- HITL 沉淀的 Golden Dataset（约 5 万对齐数据）作为种子混合
- 关键点：**故意加入失败案例**，增强模型对非理想环境的容错性

**微调技术方案**（32B 量级，全参微调显存不可行）：
```
DeepSpeed ZeRO-3 + QLoRA (nf4 量化)
LoRA 插入：q_proj, v_proj, o_proj, gate_proj, up_proj
超参数：r=64, α=128, Dropout=0.05, lr=5e-5（Cosine 退火）
硬件：8×A100 80G，FlashAttention-2 + Gradient Checkpointing
训练集：约 5 万轮多并发会话（~100M tokens），3 个 Epoch，约 14 小时
```

**收益**：建立对 `RSRP`、`PUSCH`、`BWP` 等术语的深度概念映射，将"思考过程（Scratchpad）+ 严格 JSON 工具调用语法"的肌肉记忆烙印到模型权重中。

### 5.2 DPO 偏好对齐（抑制危险幻觉）

**问题**：SFT 只能教模型"什么是正确的"，无法教它"什么是危险的边界"。

**数据采集**：HITL 组件是天然的 DPO 数据采集器：
```
高危用例被专家打回  →  Y_rejected（负样本）
专家修改后通过      →  Y_chosen（正样本）

三元组：(Prompt, Y_chosen, Y_rejected) → DPO 训练
```

**DPO 相比 RLHF 的优势**：省去独立 Reward Model 的训练复杂度，直接通过偏好对数据进行梯度对齐，工程实现更轻量。

**收益**：对"高危不确定场景"自动触发自我怀疑（Confidence Score 跌出阈值），灾难性幻觉生成率显著降低，系统具备真正的企业级安全对齐能力。

## 六、典型场景：5G 切换失败综合诊断

**用户输入**："近期配置变更后，怀疑基站 A 向基站 B 的 Xn 接口切换有问题，请验证并给出结论。"

**Step 1 — 意图解析与用例生成（Planning）**
- Agent 调用 `Test_Case_Query`，检索《3GPP TS 38.331 Xn 切换信令流程规范》及历史切换失败案例
- Qwen3-32B 生成结构化 JSON 仿真控制脚本：定义终端移动速度、A3 事件触发阈值、并发 100 次切换测试

**Step 2 — HITL 拦截（Guardrail）**
- Guardrail 节点扫描 JSON 脚本，发现频段参数被错误配置为受限频段（幻觉）
- LangGraph `interrupt_before` 触发，State 序列化至 Postgres，Celery 推送钉钉告警
- 专家修改为合法实验频段后 Approve，Agent 从断点恢复继续执行

**Step 3 — 自动化执行（Acting）**
- Agent 调用 `Simulation_Runner` 触发仿真平台执行 100 次切换
- 长时任务交由异步进程监控，Agent 主线程挂起等待回调

**Step 4 — 双轨判定（Reasoning）**
- **统计轨**：`Metrics_Collector` 拉取 KPI，切换成功率 85%（低于基线 99%），统计轨直接判 FAIL
- **语义轨**：`Log_Analyzer` 抓取 15 次失败的 PCAP 信令包，LLM 发现 `SN Status Transfer` 阶段出现丢包超时——根因不在空口，而在 Xn 回传链路

**Step 5 — 结论输出与知识回流（Reflection）**
- 输出报告："FAIL。根因：基站 A 与基站 B 的 Xn 回传链路存在拥塞丢包，建议排查承载网络，非空口问题。"
- 专家确认为 Golden Case，自动 Embedding 入库，完成认知进化内闭环

## 七、工程化部署

### 高并发推理（vLLM）
基于 vLLM 部署 Qwen3-32B，开启 PagedAttention 动态显存分页调度，KV Cache 重用率大幅提升，保证在线判定模块 30+ tokens/s 的稳定生成速率。

### 长时任务异步解耦（FastAPI + Celery）
测试执行通常耗时分钟到数小时。FastAPI 接收请求后将复杂"长图执行逻辑"打包至 Celery 异步队列，用户不被阻塞，节点状态通过 SSE 长连接实时推送至前端。

## 八、评测体系

### 评测数据集构建

- **Golden Trajectories**：从专家历史操作日志中提取标准"工具调用序列"，作为 Planning 和 Acting 阶段的 Ground Truth
- **对抗性注入**：故意加入"假失败"（网络瞬断恢复）、"模糊诉求"、"工具故障（500 报错）"，测试 Agent 的冗余纠错能力
- **动态扩充**：每日自动抓取被 HITL 触发的边缘用例，打包进回归集，评测难度随系统演进自动增加

### 四维评估矩阵

| 维度 | 指标 | 评估方法 |
|:---|:---|:---|
| **格式合规** | 指令遵从率（JSON Schema 合法性）| pytest + Pydantic Validators |
| **工具调用** | 工具选择准确率 + 参数 F1-Score | 与 Ground Truth 轨迹对比 |
| **轨迹效率** | 任务完成率 + 步数效率比（Agent步数/专家步数）| Golden Trajectories 对比 |
| **安全合规** | 高危场景阻断率（生产网必须 100%）| 毒药测试集压测 |

### 三层评估技术栈

1. **静态断言层**：pytest + Ragas/DeepEval，量化检索精度（Context Precision、Answer Relevance）
2. **LLM-as-Judge 层**：Prometheus-Eval 或 GPT-4o 对长文本报告进行语义评分，5 分制定制评分 Prompt
3. **沙盒执行层（最关键）**：Agent 生成的用例直接在 Docker 隔离环境中运行，以执行结果而非文本描述为最终判定依据，沙盒会随机注入截断日志、偶发延迟等噪声

## 十、面试高频考点

### Q1：为什么选 LangGraph 而不是基础的 LangChain AgentExecutor？

传统 `AgentExecutor` 本质是一个死循环黑盒，对通信自动化测试这种"出错就可能连锁崩溃"的任务太不可控。

LangGraph 将流程抽象为**状态机**，带来三个核心优势：
1. **容错性**：可以明确定义异常重试路径，并在图中挂载 HITL 断点
2. **多分支并发**：统计基线比对与 LLM 语义分析可作为并行节点同时激活
3. **长期记忆持久化**：原生 Checkpoint 机制，数小时长线测试任务被意外中断后可从断点恢复

### Q2：Qwen3-32B 在通信领域的幻觉问题怎么解决？

三层递进方案：
- **第一层（RAG 约束）**：Prompt 中注入模板化的历史规范字典，提供协议依据
- **第二层（SFT）**：约 5 万条 Golden Dataset 微调，把结构化格式和术语映射烙印到权重里
- **第三层（DPO）**：用 HITL 沉淀的偏好数据对齐安全边界，抑制危险幻觉

### Q3：混合检索中 BM25 和向量检索怎么权衡？

不是固定比重（如 50/50），而是**自适应调权**。

通信领域 OOV 问题严重，`PRACH`、`BWP` 等专有名词纯靠 Embedding 容易语义漂移。自研 RSF 融合算法按问题长度动态调权：短查询（精确名词匹配）偏向 BM25，长查询（意图语义）偏向向量。最终统一经 Cross-Encoder Reranker 交叉打分，截取最精准的 Top-K。

### Q4：置信度阈值 0.65 怎么来的？

通过压测找到**相变点**——低于 0.65 时 Agent 自主执行错误率急剧上升，不是拍脑袋定的。实操中阈值需要随领域数据积累持续校准。

### Q5：HITL 在工程上如何不阻塞线程？

关键是 LangGraph 的 `interrupt_before` + Postgres Checkpoint：触发 HITL 时，State 快照序列化写入 Postgres，进程终止释放 GPU 显存；Celery 异步推送钉钉告警；专家 Approve 后 API 唤醒，从 Postgres 重建 State 调用 `.resume()` 继续执行。整个等待期间无任何资源占用。

### Q6：系统需要后训练吗？怎么做的？

必须做。通用模型是"学识渊博但没有领域经验的实习生"，在严肃的 5G 测试场景中，格式错乱和危险幻觉是不能容忍的。

**SFT**：解决格式和术语，数据来自 Self-Instruct 合成 + HITL Golden Dataset，DeepSpeed ZeRO-3 + QLoRA 微调 Qwen3-32B。

**DPO**：HITL 是天然的偏好数据采集器，被打回的高危用例为负样本，专家修正后为正样本，直接绕过 Reward Model 进行偏好对齐，比 RLHF 工程实现轻量很多。

整套"执行 → 拦截打回 → 专家批改 → SFT/DPO 迭代"的闭环，使 Agent 的安全意识持续向顶级专家经验收敛。

### Q7：如何客观评估长链路 Agent 系统？

不能用单一 Accuracy，需要四个维度：
1. **格式合规**：JSON Schema 硬性校验
2. **工具调用**：工具选择准确率 + 参数 F1-Score
3. **轨迹效率**：完成率 + 步数效率比（Agent 步数 / 专家步数，防止"绕弯子成功"）
4. **安全合规**：毒药测试集阻断率

三层评估技术：静态断言（pytest）→ LLM-as-Judge（报告语义评分）→ 沙盒执行（结果优先于描述）。
