# 大模型微调：SFT (Supervised Fine-Tuning) 核心技术解析

在大语言模型（LLM）的训练流中，如果说 **预训练（Pre-training）** 赋予了模型“认识世界”的基础常识，那么 **有监督微调（Supervised Fine-Tuning, SFT）** 则是教会模型“如何与人类对话”的关键步骤。

SFT 是大模型指令遵循（Instruction Following）能力的最核心来源，也是目前企业级应用落地最常用、门槛最低的模型定制手段。

---

## 1. 什么是 SFT？与预训练的区别

预训练阶段（Pre-training），模型进行的是**无监督的“海量文本接龙”**：它读尽了维基百科、网页论坛、代码库，但如果你输入“请帮我写一封请假条”，它可能会继续“接龙”输出一段关于请假条格式的维基百科定义，而不是真去帮你写信。

**SFT（有监督微调）则是**：
通过人工（或更强模型）构建高质量的 **(指令, 回复)** 配对数据集，让模型学习在特定的提示词下，应该给出什么样的结构化、符合逻辑的回答。

**类比**：
- 预训练 $\rightarrow$ 读遍全球图书馆，成为了一个上知天文下知地理但思维跳脱的“学者”。
- **SFT $\rightarrow$ 学习“客服礼仪”与“应答规范”，成为一个能听懂人话的好助手**。
- 对齐（RLHF/DPO） $\rightarrow$ 学习人类价值观，避免说错话、教唆犯罪（成为一个安全的助手）。

---

## 2. SFT 的训练数据长什么样？

SFT 的训练数据极其注重**“质量重于数量”**。典型的一条数据结构如下（以 OpenAI 格式为例）：

```json
{
  "messages": [
    {"role": "system", "content": "你是一位拥有资深经验的儿科医生。"},
    {"role": "user", "content": "我的宝宝今天有点轻微咳嗽，并且体温 37.5 度，我该怎么办？"},
    {"role": "assistant", "content": "您好。宝宝目前的情况属于低烧伴有轻微咳嗽。首先...（详细建议）"}
  ]
}
```

这些数据通常包含多轮对话、多种复杂指令（如摘要总结、代码生成、角色扮演、格式转换等）。在训练时，模型会像学人说话一样，强行模仿大段的高质量“Assistant”的回复。

---

## 3. SFT 的主流实现手段

对于一个几百亿（甚至更大）参数规模的模型，普通的微调（全量微调 Full Fine-Tuning）非常吃显存，动辄需要几台 A100/H100 组成的集群。因此，衍生出了各种 **PEFT（参数高效微调，Parameter-Efficient Fine-Tuning）** 技术：

### 1. **全参微调 (Full Fine-Tuning，FFT)**
- **原理**：直接将模型的所有权重层解冻，使用 SFT 数据进行完整的反向传播和梯度更新。
- **优点**：模型能力的改变最为彻底，上限最高（适合从开源底座模型向极度垂直的领域迁移，如纯代码模型、医疗垂类模型）。
- **缺点**：成本极高，极度吃显存。可能产生“灾难性遗忘”（忘了预训练通用知识）。

### 2. **LoRA (Low-Rank Adaptation)** （最主流的工业界方案）
- **原理**：在原有的庞大 Transformer 模型旁边“旁路”插入两个低秩矩阵（小矩阵）。原模型权重“冻结”不更新。训练时，只更新这两个小矩阵的权重。推理时，将这两个小矩阵的权重加回原矩阵中。
- **优点**：**算力要求极低**，一个原本需要 8 张 A100 的 8B 模型，通过 LoRA 在单张 4090 上可能就能微调。而且可以挂载多个不同场景的 LoRA 权重，即插即用（拔插不影响底座模型）。
- **局限**：模型整体能力上限不如全量微调，尤其是当你试图让模型学习大量全新的事实性知识（而不仅仅是特定说话格式）时，LoRA 的表现往往不佳。

### 3. **QLoRA (Quantized LoRA)**
- **原理**：在 LoRA 的基础上更极端，将巨大的底座模型压缩（量化）到 4-bit（极大地节省了显存加载），然后再在外挂小的 LoRA 权重层上使用 16-bit 精度进行训练。
- **优点**：在极低显存（甚至是消费级 24G 显卡微调 14B 参数模型）的情况下完成了原本不可能的训练。
- **适用场景**：个人开发者、极其拮据的企业边缘侧定制。

---

## 4. SFT 最怕什么：灾难性遗忘 (Catastrophic Forgetting)

**症状**：原本一个通用模型，用大量特定领域的专业数据 SFT 了一次之后。虽然领域知识变强了，但突然不会算 1+1 了，也不会用中文正常聊天了。
**原因**：过拟合了那批微调数据，或者原有的通用权重分布被严重破坏。
**解决方案**：
- 严格控制学习率（Learning Rate不能太大）和训练步数（Epoch 数一般不要超过 3）。
- **混合通用数据集**：在特定领域的数据集里，按照比例掺杂进一部分通用对话、代码生成等高质量数据一起 SFT。

---

## 5. SFT 到底教给了模型什么？（LIMA 的启示）

有一篇极具影响力的论文叫做 [**LIMA: Less Is More for Alignment**]。这篇论文做了一个震撼的反常识实验：

他们不需要成百上千万的对话数据，**仅仅精心人工手写了 1000 条** 极其高质量、严谨排版、没有错别字的 prompt/response 数据，然后对 LLaMA 底座进行 SFT。
结果：超越了许多使用了几十万条开源由于低质量数据灌凑微调出的模型。

**核心结论（业界共识）**：
1. **大模型的知识水平，是在预训练阶段（Pre-training）决定的**。
2. **SFT 并不（或者说极难）教给模型全新的知识。SFT 的作用，是“解锁”大模型在预训练里学到的知识**，并且规定它必须以特定的格式（例如分点、加粗、特定的 JSON 形式）把知识展现出来。

## 6. SFT 落地常见陷阱与建议

1. **垃圾进，垃圾出 (Garbage In, Garbage Out)**
   如果你用弱智的吧友回答、或者带有极其明显机器味的低质量 GPT-3.5 数据去 SFT 基座模型，最后出来的模型一定也是弱智和充满了极其刻板的特定句式（比如动不动就说“作为一个人工智能助手...”）。
2. **数据多样性**
   如果在客服训练集中所有问题都是“退货问题”，即使你准备了十万条，模型最后遇到“我要修改密码”也会回答成如何退货。
3. **“强迫症”格式调优优先**
   大多数企业的落地场景中，使用 SFT 的最主要目的**不是**让它更聪明，而是**固定其输出格式（比如强制输出强规则排版的 JSON或者特定 markdown 表格）**。在这个诉求上，提供几百条优质的 SFT 训练集远胜过每天绞尽脑汁修改提示词Prompt。
